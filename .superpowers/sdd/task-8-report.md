# 작업 8 구현 결과 보고서

상태: 7개 업무가 함께 사용하는 영속 알림 설정 화면과 서버 기준 가용성 경계를 구현하고 로컬 검증을 완료했습니다.

구현 커밋: `ea8e1fc`

## 구현 범위

- 할 일, 영어 단어 재시험, 등록, 전반, 퇴원, 휴보강, 전자결재를 서버 레지스트리 순서로 제공하는 공통 알림 설정 화면을 만들었습니다.
- 전역 설정 페이지와 업무별 대화상자가 같은 조회·초안·명시적 저장 API를 사용합니다. 업무별 화면은 자체 기본값이나 별도 저장소를 갖지 않습니다.
- 데스크톱에서는 업무 행렬, 모바일에서는 같은 초안으로 만든 카드 목록을 사용하며 불가능한 대상·채널 조합은 표시하지 않습니다.
- 저장, 저장 후 알림 재계산, 재계산 완료·실패·재시도를 서로 다른 상태로 분리했습니다. 재계산 재시도는 설정 저장을 반복하지 않습니다.
- 동일 초안의 모호한 저장 재시도에는 같은 요청 ID를 유지하고, 초안이 바뀌면 이전 요청 ID와 오래된 재계산 작업을 폐기합니다.
- revision 충돌은 최신 설정 불러오기와 내 변경 유지로 나눴습니다. 같은 항목 덮어쓰기는 별도 요청 ID와 충돌 필드 감사 정보를 요구하며, 확인 후 초안을 다시 바꾸면 재확인 상태를 복원해 감사 우회를 막습니다.
- 닫기, ESC, 바깥 클릭, 탭·링크 이동, 뒤로 가기, 새로고침을 하나의 저장하지 않은 변경 확인 흐름으로 통합했습니다. 저장 중에는 이동 결정을 실행하지 않습니다.

## 연결과 가용성

- Google Chat 연결은 규칙 설정과 분리된 `연결` 영역에서 확인합니다. 직원은 상태만 보고, 관리자만 교체·검증·해제할 수 있습니다.
- Webhook 저장은 테스트 메시지를 자동 발송하지 않습니다. 연결 검증은 별도의 확인 동작이며 브라우저 자동 검증에서는 실제 메시지를 보내지 않았습니다.
- 서버 플래그와 `common_notification_control_plane_runtime_version() = 1`이 모두 준비된 경우에만 공통 설정을 엽니다.
- 플래그가 정상적으로 꺼져 있으면 기존 화면을 유지하고, 권한·서버·마커 상태를 확인할 수 없으면 기존 화면과 공통 화면을 모두 숨기는 실패-폐쇄 상태를 표시합니다.
- 7개 업무의 닫힌 설정 레지스트리와 휴보강 영속 설정의 멱등 import를 추가했습니다. 모든 dispatch 플래그는 계속 `false`입니다.
- 휴보강 import는 저장 원본별 정확한 규칙 수, 활성값, 활성 템플릿, source revision·checksum을 보존합니다. 전송 이력·발생 건·소유권은 이 단계에서 가져오지 않습니다.

## 외부 상태와 남은 배포 전 증거

- Supabase 플러그인은 읽기 전용 확인에만 사용했습니다. Docker를 요구하거나 실행하지 않았습니다.
- 원격 마이그레이션 적용, 원격 데이터 변경, 런타임 플래그 변경, 실제 테스트 메시지·provider 발송, 배포는 수행하지 않았습니다.
- 현재 원격 DB에는 작업 8의 최종 런타임 마커와 플래그 테이블이 아직 없으므로 로컬 브라우저는 의도한 한글 실패-폐쇄 화면을 표시합니다.
- 7개 패널의 실제 DB 저장·닫기·재접속·전역/업무별 동일 revision 확인과 pgTAP 실행은 마이그레이션 적용이 허용된 local/preview DB의 배포 전 게이트로 남겼습니다. 이를 실행한 것으로 보고하지 않습니다.
- 등록·전반·퇴원·휴보강 등의 활성 sender가 저장된 canonical 규칙을 실제 소유하는 것은 후속 작업 14~21의 범위입니다. 그 전까지 운영 UI 플래그는 켜지 않습니다.
- 작업 트리 전용 개발 서버는 `http://localhost:3001`에서 계속 실행 중입니다.
