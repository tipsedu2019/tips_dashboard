# 작업 12 구현 결과 보고서

상태: 등록 예약 충돌 초안 보존, 변경 전후 확인, 공통 알림 재계산 상태 조회·재시도, 실행 준비 상태의 실패 폐쇄 처리를 구현하고 독립 검토까지 완료했습니다.

코드 커밋: `2d45756` (`feat: harden registration appointment editing`)

## 예약 충돌과 요청 멱등성

- 409 충돌이 발생해도 사용자가 입력한 일시·장소·과목·남은 예약 교체 여부를 별도 초안으로 보존합니다.
- `최신 예약 비교`, `다시 적용`, `계속 편집`을 구분했습니다. 최신 서버 리비전은 화면을 다시 불러온 뒤 갱신되며, 로컬 초안은 사용자가 명시적으로 다시 적용할 때만 새 리비전 위에 재기반합니다.
- 논리적으로 같은 저장 요청은 같은 요청 키를 유지합니다. 초안이 실제로 바뀌거나 검토 후 재기반한 경우에만 새 요청 키를 만듭니다.
- 예약 취소 시 운영자가 임의 사유를 만들어 입력하지 않도록 필수 취소 사유 입력란을 제거하고, 기존 RPC에는 호환값 `reason: ""`만 전달합니다.

## 확인과 알림 재계산

- 저장·취소 전 확인창은 이전 예약, 이후 예약, 과목, 미래 알림 회차 변화를 함께 보여 줍니다.
- 예약 변이 응답에서는 공통 알림 작업의 종류와 ID만 보존하고 내부 이벤트·payload·대상 정보는 화면에 노출하지 않습니다.
- 상태 확인은 `get_notification_orchestration_job_status_v1`, 실패 재시도는 `retry_notification_orchestration_job_v1`만 호출합니다. 예약 저장·취소 RPC를 다시 실행하거나 비공개 작업 행을 직접 만들지 않습니다.
- 작업 종류, workflow, 상태, 시도 횟수를 폐쇄형으로 검증하고 알 수 없는 값은 실패 폐쇄합니다.
- 작업이 모두 성공한 경우에만 `알림 재계산 완료`라고 표시합니다. 상태를 확인하지 못한 경우에는 저장 완료와 알림 상태 미확정을 분리해 안내합니다.

## 실행 준비 상태 안전 게이트

- 등록 리마인더 런타임 마커와 공통 어댑터 런타임 마커가 모두 버전 1이고, Worker·감시자 성공 heartbeat가 모두 3분 이내일 때만 처리 상태 조회와 재시도를 엽니다.
- 화면은 인증 토큰으로 고정된 `GET /api/notifications/operations?view=registration-processing-readiness`를 `no-store` 방식으로 60초마다 확인합니다.
- 작업 20의 운영 API와 최종 런타임이 설치되기 전에는 조회 실패를 삼키고 안전하게 기능을 숨깁니다. 브라우저 값만으로 준비 상태를 만들지 않습니다.
