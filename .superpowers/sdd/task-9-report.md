# 작업 9 구현 결과 보고서

상태: 사용자별 알림 읽음 상태와 현재 브라우저 기준 Push 준비 상태를 구현하고, 코드·브라우저·프로덕션 빌드 검증을 완료했습니다.

구현 커밋: `f3fbf26`

## 알림함 구현

- 알림 목록, 미확인 수, 읽음 처리를 `get_dashboard_notification_inbox_v1`, `get_dashboard_notification_unread_count_v1`, `mark_dashboard_notification_read_v1` 세 RPC로 통일했습니다. 브라우저는 사용자 ID를 보내거나 알림 테이블을 직접 읽지 않습니다.
- 목록·개수·읽음 처리가 같은 비공개 가시성 관계를 사용하도록 추가 마이그레이션을 만들었습니다. 내부 등록 상담 Chat claim은 알림함에서 제외됩니다.
- 읽음 처리는 현재 로그인 프로필의 receipt만 기록하며 공유 알림 행의 `read_at`을 변경하지 않습니다. 과거 `read_at`은 개인 receipt가 없는 기존 행의 표시 호환에만 사용합니다.
- 읽지 않은 항목의 `읽음` 버튼을 링크 바깥 형제로 분리했습니다. 버튼은 팝오버와 URL을 유지하고, 항목 링크는 이동을 막지 않은 채 읽음 요청만 먼저 시작합니다.
- 프로필 전환, 목록 새로고침, 개별 읽음 처리, 미확인 수 재동기화가 겹쳐도 오래된 응답이 최신 상태를 덮지 않도록 프로필·세대·스냅샷 경계를 추가했습니다.

## Push 준비 상태 구현

- 브라우저 API, 보안 연결, 서버 VAPID 설정, 권한, 서비스 워커·manifest, 현재 구독, 현재 프로필 소유권, 고정 자가진단 결과를 순서대로 판정합니다.
- 권한 요청, 연결, 다시 연결, 고정 자가진단, 해제는 사용자의 명시적 동작에서만 실행됩니다. 상태 확인만으로 권한 요청이나 실제 Push 전송이 발생하지 않습니다.
- Push 구독 삭제 API는 현재 프로필이 실제로 삭제한 행의 존재를 `deleted`로 반환합니다. 명시 해제와 공개키 교체 모두 `deleted=false`이면 로컬 브라우저 구독·새 구독·POST를 건드리지 않습니다.
- 만료 구독 정리 실패와 감사 기록 실패를 전송 결과와 분리했습니다. 이미 확정된 전송·만료 결과를 실패로 바꾸지 않으며, 서버 정리 실패 원인도 감사 경고에 가려지지 않습니다.
- 프로필 전환은 진행 중인 Push 작업을 중단하고, 포커스·가시성·팝오버 열기 새로고침은 사용자 작업 중 상태를 취소하지 않습니다.

## 현재 실행 상태

- 작업 트리 전용 개발 서버는 `http://localhost:3001`에서 계속 실행 중입니다.
- 실제 화면에서 알림 팝오버를 열어 새로고침과 실패 상태 문구를 확인했습니다.
- 원격 DB에는 이번 추가 RPC와 런타임 마이그레이션이 아직 적용되지 않았으므로 현재 화면은 `알림을 불러오지 못했습니다. 다시 시도하세요.`를 정확히 표시합니다.
- 서버 VAPID 키와 연락처도 현재 로컬 실행 환경에 완비되지 않아 `서버 Push 키와 연락처 설정을 확인해야 합니다.`를 표시합니다. 성공으로 가장하지 않습니다.

## 외부 상태

- Supabase 플러그인은 원격 스키마와 마이그레이션 상태를 읽는 데만 사용했습니다. Docker는 요구하거나 실행하지 않았습니다.
- 원격 마이그레이션·데이터·플래그 변경, 배포, 실제 Push 자가진단 발송은 수행하지 않았습니다.
- 실제 DB pgTAP과 두 프로필 receipt 영속성 확인은 마이그레이션이 적용된 승인된 로컬 또는 미리보기 DB에서 실행해야 하며, 이번 로컬 소스 검증에서 실행한 것으로 보고하지 않습니다.
