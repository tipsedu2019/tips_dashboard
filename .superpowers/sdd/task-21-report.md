# 작업 21 구현 결과 보고서

상태: 운영 상태를 바꾸지 않는 알림 전환 검증기와 결정적 그림자·부분/전체 롤백 고정 검증 자료 구현을 완료했습니다. 전체 출고 판정은 작업 22에 기록합니다.

## 읽기 전용 전환 검증

- 승인된 런타임 플래그 12개와 소유자 전환 순서 10개를 정확한 상수로 고정했습니다.
- 공통·업무 어댑터·등록 실행 표식, 작업 처리기 중단 래치, 작업 처리기·감시기 3분 정상 동작 신호, 대기 지연과 즉시 중단 지표를 실패 폐쇄 방식으로 판정합니다.
- 기존·정규 의도를 업무, 이벤트, 발생 ID, 대상 그룹, 채널, 대상 키, 수신자 세대, 템플릿 체크섬, 정규화 렌더 해시로 비교합니다.
- `targetGeneration`은 10진 문자열인 업무 수신자 세대로만 인정하며 별도 `ownerGeneration`으로 대체하면 실패합니다.
- 검증 스크립트는 네트워크, RPC, 플래그·스케줄·DB 변경 코드를 포함하지 않습니다.

## 그림자와 소유권 고정 검증 자료

- 10개 소유자 각각에 결정적 검증 주기를 두고 기존 경로는 주입 기록기로 의도만 기록합니다.
- 정규 경로는 현재 템플릿 초기값과 실제 `notification-workflow-registry` 어댑터, `notification-worker.renderNotificationSnapshot`을 사용합니다. 기존 경로는 정규 템플릿 객체나 체크섬 함수를 공유하지 않고 문구·치환값·링크·대상·직렬화를 별도 구현으로 계산합니다.
- 정규 대상만 바꾼 회귀가 실제 불일치로 검출되므로 같은 값을 양쪽에 넣는 자기비교가 아닙니다.
- 정규 그림자 행은 모두 `skipped/shadow_mode`, 재실행 불가로 끝나며 외부 요청, 알림함 투영, 중복 요청은 0건이어야 합니다.
- 운영 전환 완전성은 모든 활성 규칙의 현재 자연 발생 비교로만 판정합니다. 결정적 결과는 운영 자연 비교를 대신하지 않으며, 활성 규칙이 0개인 범위만 현재 상태에 묶인 무활성 증거를 인정합니다.
- 저빈도 결정적 검증은 순수 로컬 코드·미리보기 추가 증거로만 유지합니다. 운영 DB에서 결정적 준비·기록 RPC를 호출할 권한은 서비스 역할에도 주지 않습니다.
- 동시에 여러 결정적 계산이 실행돼도 외부 요청 차단이 먼저 풀리거나 전역에 남지 않게 했습니다.
- 새 템플릿 체크섬은 SHA-256으로 강제하고 과거 32자리 체크섬은 읽기 호환합니다. 사용자 편집 템플릿의 실제 일치는 운영 자연 발생 비교에서 검증합니다.
- 전환 전 단계와 부분 롤백 뒤 모든 업무에서 기존 또는 정규 소유자가 정확히 한 명인지 검증합니다.
- 그림자 중 공급자 또는 알림함 부작용이 감지된 검증 자료는 12개 플래그를 모두 끄고 작업 처리기 중단 래치를 올리며 그림자를 자동 복구하지 않습니다.

## 원자 롤백 고정 검증 자료

- 부분 롤백은 대상 소유자만 끄고 다른 정규 소유자와 전역 그림자 꺼짐 상태를 보존합니다.
- 전체 소유자 롤백만 그림자 복구를 허용합니다.
- `pending`·`retry_wait`는 취소하고 전송 전 `claimed`에는 취소 요청을 남기며, `sending`·`sent`·`failed`·`delivery_unknown`과 공급자 참조는 보존합니다.
- 소유권 이전은 점유 종료 뒤 예약·미전송·공급자 참조 없음 상태에서만 허용합니다. 시작됨, 전송 중, 전송 완료, 공급자 참조 있음, 전달 결과 불명은 거절합니다.
- 알 수 없는 소유자, 혼합 소유자, 오래된 예상 변경 번호, 부분 롤백의 그림자 복구 요청은 실패하며 입력 플래그·전달·취소 요청·소유권이 그대로 남습니다.
