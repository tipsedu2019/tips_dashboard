# 알림 업무 전환·롤백 운영 절차

이 문서는 공통 알림 코드와 미리보기 검증이 끝난 뒤 사용할 미래 전환의 안전 요구조건 기록이다. 현재 실행 가능한 전환 절차나 운영 승인서가 아니다. 검증 스크립트는 읽기 전용이며 운영 플래그를 변경하지 않는다. 배포, 마이그레이션 설치, 일정 설치, 운영 그림자 기록, 소유자 전환은 각각 범위가 명시된 별도 승인이 있어야 한다.

## 격리 전환 정책

quarantine의 과거 6개 SQL은 reference-only이며 직접 적용하거나 active lane으로 승격하지 않는다. 관찰을 다시 시작해도 먼저 최신 schema 기준의 새 forward-dated install migration과 service-role 전용 activation RPC를 별도 설계·검증·승인한다.

과거 SQL은 현재 과학 인지 구현인 `public.revalidate_immediate_notification_delivery_v1`와 `public.prepare_notification_immediate_delivery_v1`를 과학 지원 이전 본문으로 덮어쓸 수 있다. 따라서 아래의 24시간 이상·Asia/Seoul 기준 완결된 하루 관찰과 7일 운영 shadow는 미래 전환에도 유지할 역사적 안전 요구조건이지만, 충족 여부가 quarantine SQL의 적용·복사·이름 변경·승격 권한을 만들지는 않는다.

독립 보안 점검에서 발견된 prepare 함수의 PUBLIC/브라우저 EXECUTE 결함은 quarantine 활성화와 무관한 `20260722130000_notification_prepare_acl_hardening.sql` 한 건으로만 보정한다. 이 active migration은 함수 본문·데이터·runtime flags·secret·cron·provider를 바꾸거나 전환을 시작하지 않는다. 정식 적용 뒤 정확한 시그니처의 기대 ACL은 postgres owner, `service_role EXECUTE=true`, `anon/authenticated EXECUTE=false`, PUBLIC 직접 grant 없음, service-role grant option 없음이다. 미래 install/activation 설계도 이 ACL을 넓히거나 과학-aware 본문을 교체할 수 없다.

## 전환 전에 반드시 확인할 것

- 런타임 플래그 레지스트리는 승인된 12개 행과 정확히 일치해야 한다.
- 공통·업무 어댑터 런타임 표시는 모두 버전 1이어야 한다. 등록 관련 소유자를 다룰 때는 등록 예약 알림 런타임 표시도 버전 1이어야 한다.
- 비공개 작업 처리기 중단 래치는 해제 상태여야 한다.
- 작업 처리기와 독립 감시기의 마지막 성공 정상 동작 신호가 모두 3분 이내여야 한다.
- 그림자 모드의 실제 공급자 요청과 정규 알림함 투영은 각각 0건이어야 한다.
- 중복 외부 시도, 새 전달 결과 불명(`delivery_unknown`), 5분 초과 대기, 범위 불일치, 대상 없는 활성 규칙, 소유권 이상, 그림자 의도 불일치는 모두 0건이어야 한다.
- 비교의 `targetGeneration`은 10진 문자열인 업무 수신자 세대다. 별도 발송 소유자 세대인 `ownerGeneration`으로 대체하면 검증 실패다.

## 구형 계약 관찰과 폐쇄

이 절차는 Docker를 사용하지 않는다. 아래 내용은 과거 브리지 설치와 미래 재설계에 남겨야 할 관찰 요구조건이며 현재 실행 항목이 아니다. 이미 설치된 브리지 SQL을 재실행하지 않고, quarantine SQL도 어떤 도구로도 적용하지 않는다. 미래 설치·활성화의 실제 명령과 승인 경계는 새 forward-dated install migration과 별도 service-role activation RPC가 준비된 뒤 새 운영 계획에 작성한다.

폐쇄 전 호환 번들은 Google Chat과 Web Push의 구형 원문 호출을 오류로 만들지는 않지만 외부 공급자에는 보내지 않는다. 인증된 호출에 한해 진입점, 계약 종류, 결과, 행위자, 임의 요청 ID만 비공개 계측표에 남기고 `skipped=true`, `reason=legacy_payload_observed`로 끝낸다. 같은 업무 변경에서 생성된 안정된 원본 이벤트 ID를 서버의 고정 경로가 해석해 실제 전송을 담당하며, 브라우저가 보낸 문구와 대상을 사용하지 않는다. 제목, 본문, 수신자, 링크, 웹훅, 구독 주소 같은 내용은 계측표에 저장하지 않는다.

### 역사적 1단계 기록: 브리지 설치와 관찰

1. 과거 1단계에서는 다음 active-lane 브리지 마이그레이션이 순서대로 설치됐다. 이 목록은 재적용 지시가 아니라 관찰 전제의 기록이다.
   - `20260716190000_notification_ops_task_producers.sql`
   - `20260716191000_notification_transfer_withdrawal_producers.sql`
   - `20260716192000_notification_makeup_adapter.sql`
   - `20260716193000_notification_approval_producers.sql`
   - `20260716194000_notification_registration_handoffs.sql`
   - `20260716194500_notification_legacy_contract_telemetry.sql`
2. 과거 1단계는 위 DB 계약과 같은 커밋의 애플리케이션을 Vercel 운영 환경에 배포했다. 194500의 전달 의도 RPC는 그림자 기능이 아직 없는 관찰 단계에서 검증된 `recorded=false`, `reason=shadow_contract_pending`만 반환했다. quarantine의 195900 본문은 당시 같은 함수 서명을 실제 그림자 비교 구현으로 교체하려던 reference이며, 현재 적용 순서나 승격 지시로 사용하지 않는다.
3. 과거 관찰 증거는 배포 영수증 기록기를 5분 주기로 운영하고, Vercel 운영 배포 ID·프로젝트 ID·실행 중 계약 해시를 대조한 뒤 일치한 배포 ID 해시와 `buildRevisionHash`만 DB에 기록하는 모델이었다. 당시에도 호출자가 서버 수를 제출하지 않았고, 비밀정보는 파일이나 로그에 남기지 않는 요구조건이었다.
4. 과거 증거 모델은 24시간 이상과 Asia/Seoul 기준 완결된 달력일 하나를 동시에 포함하고, 모든 영수증이 하나의 `buildRevisionHash`를 가리키며 인접 영수증 간격이 10분을 넘지 않아야 했다. 중간 배포가 있으면 새 빌드 기준으로 관찰 창을 다시 시작하고, `ops-task`와 `makeup` 고정 경로의 실제 성공을 각각 한 건 이상 요구했다. HTTP 2xx여도 `failed`가 1 이상이거나 `sent + deduped`가 0이면 증거가 아니었다.

위 두 항목은 과거 관찰 증거의 설명일 뿐 지금 실행할 스크립트·스케줄·자격증명 주입·DB 기록·실제 업무 exercise 지시가 아니다. 최신 schema 기준의 새 forward-dated install migration과 별도 service-role 전용 activation RPC가 설계·검증·승인되기 전에는 service-role 자격증명을 주입하거나 배포 영수증을 DB에 쓰거나 `ops-task`·`makeup` 실제 exercise를 수행하는 등 DB-writing 관찰을 재개하지 않는다. 새 계획이 승인되더라도 24시간 이상 및 Asia/Seoul 기준 완결된 하루와 이후 7일 운영 shadow 게이트는 유지한다.

과거 Vercel 영수증은 운영 도메인이 가리키는 서버 배포의 교체 완료를 증명했다. 열려 있던 오래된 브라우저 탭의 존재를 추정해 100% 같은 숫자를 만들지는 않았고, 브라우저 잔존 여부는 24시간 동안 들어온 실제 구형 계약 트래픽과 원본 ID 변환 실패가 모두 0인지로 판단했다. 이 안전 판정 계약도 미래 설계에서 유지하되 현재 관찰 재개 명령으로 해석하지 않는다.

### 역사적 증거 판정 계약

과거 설계는 읽기 전용 `public.get_notification_contract_drain_evidence_v1` 결과를 가공하지 않고 보존한 뒤 로컬 `scripts/verify-notification-contract-drain.mjs`로 판정했다. 이 이름들은 요구조건 추적을 위한 reference이며 현재 호출 명령이 아니다. 새 install migration과 별도 activation RPC가 승인된 뒤 새 운영 계획에서만 실제 증거 수집·검증 명령을 정의한다.

역사적 `passed: true` 계약은 구형 변환 불가 호출, 원본 ID 변환 실패, 미완료 고정 경로, 실패한 고정 경로가 모두 0이어야 했다. 두 고정 경로의 동일 빌드 성공은 각각 1건 이상이고, 배포 영수증은 구간 처음과 끝 5분 안을 덮으며 최대 공백이 10분 이하여야 했다. 모든 영수증의 활성 서버 배포는 브리지 인식 배포이고 `deploymentBuildRevisionCount`는 1, `latestCompliantBuildRevisionHash`는 64자리 소문자 SHA-256이어야 한다는 요구조건을 미래 설계에도 유지한다.

### 역사적 2단계 계약: DB 자체 재검사와 폐쇄

과거 `20260716195000_notification_workflow_legacy_closure.sql`이 표현한 안전 계약은 외부 JSON을 신뢰하지 않고 DB의 직전 Asia/Seoul 달력일 전체부터 현재까지를 직접 다시 집계하는 것이었다. 브리지 행을 잠근 상태에서 구형 트래픽, 변환 실패, 단일 빌드, 그 빌드의 고정 경로별 실제 성공·실패, 배포 영수증 연속성을 다시 확인하고, 통과한 빌드 해시와 증거 범위를 불변 표식에 저장하는 요구조건은 미래 install migration에도 유지한다. 이 reference SQL 자체는 검증 통과 여부와 무관하게 적용하지 않는다.

나머지 5개 quarantine SQL이 표현한 worker 일정, provider claim, forward-compat, shadow 증거 계약도 reference로만 보존한다. 미래 install migration은 최신 schema와 두 과학 인지 함수의 최종 본문을 기준으로 이 계약을 새로 조합해야 하며, 설치와 activation을 한 SQL에 묶지 않는다. activation은 새로 설계한 service-role 전용 RPC와 별도 승인으로만 수행하고, 자연 비교 완전성·무활성 범위 현재성·새 SHA-256 및 과거 32자리 체크섬 읽기 호환 요구조건은 유지한다.

## 그림자 비교

코드 단계의 고정 검증 자료와 모의 전송 검증은 회귀 방지용 테스트일 뿐 운영 전환 증거가 아니다. 합성 이벤트나 동일한 렌더 결과를 기존·정규 양쪽에 넣어 만든 자기비교 결과는 일치하더라도 전환 증거로 인정하지 않는다.

아래 항목을 정규화한 뒤 비교한다.

- 업무, 이벤트, 발생 ID
- 대상 그룹, 채널, 대상 키
- 수신자 세대
- 템플릿 체크섬
- 정규화된 렌더 결과 해시

차이는 누락 이벤트, 추가 이벤트, 대상 불일치, 수신자 세대 불일치, 채널 불일치, 템플릿 불일치로 분류한다. 미리보기와 테스트 고정 검증 자료에서는 기존 경로와 정규 경로를 합친 실제 외부 요청이 0건이어야 한다.

코드 회귀 검증은 아래의 순수 로컬 미리보기 실행기로 수행한다. 환경 변수, 서비스 역할 키, DB, Supabase API, 공급자 접속 주소가 필요하지 않다.

```bash
node --experimental-strip-types scripts/run-notification-shadow-preview-fixtures.mjs
```

실행기는 고정 순서의 10개 범위를 모두 실행한다. 정규 쪽은 현재 저장소의 템플릿 초기값과 등록 고정 규칙 정의를 독립 원본으로 삼아 `notification-workflow-registry`의 실제 업무 어댑터와 `notification-worker.renderNotificationSnapshot`을 실행한다. 기존 쪽은 정규 템플릿 객체나 체크섬 함수를 공유하지 않고 별도 문구·치환값·링크·대상 선언과 독립 직렬화로 주입 기록기 의도를 계산한다. 두 결과는 `compareNotificationShadowIntents`로 대조하므로 한쪽 대상·렌더링·링크·체크섬만 달라지는 회귀가 실패한다. 사용자 편집 템플릿의 실제 운영 일치는 이 로컬 자료가 아니라 7일 자연 발생 비교에서 검증한다. 새 템플릿 체크섬은 DB 트리거가 SHA-256으로 계산하고, 과거 설정 저장에서 만들어진 32자리 체크섬은 기존 전달·감사 지문을 바꾸지 않은 채 읽기 호환한다.

각 정규 결과는 실행 중 생성된 `skipped/shadow_mode`, `replayable=false` 행이어야 한다. 네트워크·공급자·알림함·DB·중복 외부 시도 건수가 모두 0이 아니면 실행은 실패한다. 성공 시 범위별 비교 결과와 의도 요약값을 JSON으로 출력하고, 최상위 `manifest.digest`에 정렬 JSON 기준 SHA-256을 남긴다. 같은 코드에서 다시 실행하면 출력과 최상위 명세가 동일해야 한다.

이 미리보기 결과는 코드 단계 회귀 증거일 뿐이며, 아래의 운영 자연 비교 또는 활성 규칙 0개 증거를 대신하지 않는다. 7일 관찰 조건은 그대로 유지한다.

운영 그림자가 시작된 뒤에는 범위당 임의 한 건이 아니라 현재 활성 규칙마다 실제 자연 발생 일치 비교가 필요하다. 규칙 ID, 규칙 변경 번호, 활성 템플릿 ID·체크섬 중 하나라도 현재값과 다르거나 활성 규칙 한 개가 빠지면 범위 전체가 미완료다. 결정적 로컬 결과는 이 운영 자연 비교를 대신할 수 없다. 자연 발생한 운영 비교에서 정규 전달은 `skipped/shadow_mode`로 끝나고 정규 알림함 반영과 정규 공급자 시작이 없어야 한다. 실제 발송 소유자인 기존 경로는 공유 소유권 원장에서 발송을 시작하고 `closed/sent`로 정상 종료해야 하며, 같은 소유권의 `legacy_dispatch_finalized/sent` 불변 감사 기록이 있어야 증거로 인정한다. 아직 `reserved`이거나 `dispatch_started`인 기존 소유권은 미종결 작업이므로 비교 증거가 아니며 전환도 막는다. 활성 규칙이 전혀 없는 범위만 아래의 별도 무활성 증거를 사용한다.

`record_notification_shadow_fixture_evidence_v1(scope, request_id)`라는 기존 RPC 이름은 운영 스크립트 호환을 위해 유지하지만, 더 이상 비교 고정 자료를 만들지 않는다. 호출자는 범위와 요청 ID만 전달하고 DB가 현재 상태를 잠근 뒤 증거 유형을 결정한다.

- 활성 규칙이 한 개 이상이면 현재 활성 규칙·템플릿과 일치하고 합성 검증 자료 원본이 아니며, 정규 전달이 `skipped/shadow_mode`, 기존 소유권이 `closed/sent`인 자연 발생 운영 비교를 규칙마다 요구한다. 한 규칙이라도 없으면 `notification_shadow_natural_comparison_required` 또는 최종 완전성 오류로 중단한다.
- 저빈도 경로의 결정적 검증은 위의 순수 로컬 미리보기 실행기에서 추가로 수행한다. 서비스 역할을 포함한 운영 역할은 `prepare_notification_shadow_deterministic_fixture_v1`과 `record_notification_shadow_deterministic_evidence_v1`을 실행할 수 없고, 결정적 결과는 운영 전환 완전성이나 재실행 성공으로 인정되지 않는다.
- 로컬 결정적 실행은 합성 업무 이벤트·전달·소유권·공급자 요청·알림함을 만들지 않는다. 할 일·결재 댓글에는 실제 댓글 원본 종류를 사용하고, 결재·휴보강의 원본 이벤트 ID와 업무 대상 ID를 분리한다. 단일값·배열에 중복과 서로 다른 수신자를 함께 넣어 병합·중복 제거를 검증하며, 휴보강 `subject_team`은 영어와 수학 연결을 서로 다른 발생 건으로 모두 실행한다.
- 활성 규칙이 0개이면 비교를 조작해 만들지 않고 `no_active_rule` 증거를 별도 표에 남긴다. 이 증거는 현재 범위 설정 요약값, 현재 그림자 변경 번호, 활성 규칙 0건에 묶인다. 규칙의 활성 여부·변경 번호·활성 템플릿·체크섬이 바뀌거나 그림자 변경 번호가 바뀌면 즉시 무효가 된다. 이후 규칙이 하나라도 활성화되면 모든 활성 규칙의 현재 자연 발생 비교가 새로 필요하다.

모든 분기는 공급자 호출과 알림함 반영을 하지 않으며, 합성 이벤트·전달·소유권 생성을 하지 않는다. 서비스 역할도 요약값이나 완료 횟수 같은 판정값을 제출할 수 없다.

10개 범위 계획만 먼저 확인한다. 이 명령은 DB를 호출하지 않는다.

```bash
node scripts/run-notification-shadow-fixtures.mjs \
  --request-id '<이번 10개 범위 묶음의 새 UUID>'
```

과거 운영 그림자 설계는 승인된 단계에서만 같은 묶음 요청 ID로 원격 증거 기록을 허용하고, 서비스 역할 키를 환경 변수로만 주입하도록 요구했다. 현재는 이를 실행하지 않는다. 새 install migration과 별도 activation RPC가 승인될 때 원격 실행 명령도 새 계획에서 다시 정의한다.

스크립트는 고정 전환 순서의 10개 범위를 한 번씩 호출하고 범위마다 서로 다른 요청 ID를 보낸다. 각 호출은 요청 원장의 완료 응답이 있더라도 현재 그림자 변경 번호·범위 설정·활성 규칙·활성 템플릿·실행 표식과 증거의 현재성을 다시 검증한다. 과거 성공이 현재와 다르면 재실행도 실패한다. 범위 결과가 `natural_comparison`이면 활성 규칙 수가 1개 이상이고 `comparisonKey`가 있어야 한다. 결과가 `no_active_rule`이면 활성 규칙 수가 0이고 `comparisonKey`는 `null`이어야 한다. 자연 비교가 필요한 활성 범위에서 현재 비교를 찾지 못하면 결정적 결과로 우회하지 않고 즉시 중단한다. 10개 범위를 처리한 뒤 DB가 모든 활성 규칙의 현재 자연 비교를 다시 확인해야 완료다. 단일 빌드 연속성은 이 실행기가 아니라 첫 소유자 전환 RPC가 배포 영수증으로 검증한다. 이 스크립트는 실제 학생·직원·예약 ID, 실제 연결값, 구독 주소, 공급자 접속 주소를 읽거나 만들지 않는다.

첫 소유자 전환 RPC는 그림자 시작 후 7일, 최근 5분의 조용한 구간, 현재 규칙 변경 번호·활성 템플릿 체크섬에 연결된 10개 범위 증거를 DB에서 다시 확인한다. 모든 활성 규칙은 그 규칙의 현재 자연 발생 비교가 있어야 하며 결정적 로컬 결과는 대신할 수 없다. 또한 그림자 시작부터 전환 시각까지 배포 영수증이 처음과 끝 5분을 덮고, 최대 공백 10분 이하이며, 전부 브리지 인식 서버이고, 폐쇄 표식과 같은 단일 `buildRevisionHash`인지 재검산한다. 전환 트랜잭션은 브리지 행과 규칙·템플릿 표를 잠가 배포 영수증·설정 저장·증거 판정을 직렬화하고, 첫 전환 시 빌드 증거 시작 시각·빌드 해시와 10개 범위 설정 요약값을 불변 감사 기록에 고정한다. 이후 소유자 전환도 최초 그림자 시작부터 현재까지 같은 연속 영수증과 같은 빌드인지 다시 확인한다. 중간에 다른 빌드가 한 번이라도 관찰되거나 설정 요약값이 달라지면 전환하지 않고 그림자 관찰부터 다시 수행한다.

## 자동 중단과 롤백

운영 전환 전 그림자 단계에서 공급자 요청이나 알림함 반영이 한 건이라도 발생하면 그림자 중단 RPC를 호출해야 한다. 모든 발송 플래그와 설정 UI를 끄고, 보내지 않은 작업을 취소하며, 작업 처리기 중단 래치를 올린다. 자동으로 그림자를 다시 켜지 않는다.

전환 시작 뒤 작업 처리기·감시기·일정·Vault·경로의 전역 장애나 업무 간 중복 시도는 전체 소유자 롤백 대상이다. 전체 소유자 롤백은 현재 켜진 모든 소유자 플래그를 한 트랜잭션에서 끄고 전역 그림자를 안전 기준인 꺼짐으로 둔다. 장애가 그림자·작업 처리기 안전을 훼손하지 않았고 모든 활성 소유자를 함께 내리는 경우에만 그림자 복구를 별도로 요청할 수 있다.

한 업무의 지연, 전달 결과 불명, 대상 없는 규칙, 범위·소유권 이상, 그림자 불일치는 부분 롤백 대상이다. 부분 롤백은 해당 소유자만 끄고 그림자를 계속 끈 상태로 유지하므로 정상인 다른 정규 소유자는 계속 동작한다.

롤백은 `pending`·`retry_wait`를 취소하고 보내기 전 `claimed`에는 취소 요청을 남긴다. `sending`, `sent`, `failed`, `canceled`, 전달 결과 불명 기록은 보존한다. 전송이 시작됐거나 공급자 참조가 있는 소유권은 기존 경로로 넘기지 않는다. 롤백 실패는 치명 경보로 기록하고 성공 정상 동작 신호를 남기지 않으며 다음 전환을 금지한다.

### 정규 경로에서 기존 경로로 소유권을 반환하는 절차

플래그 롤백 성공은 개별 발송의 소유권 반환 완료가 아니다. 아래 절차가 모두 끝나기 전에는 해당 발송을 기존 경로에서 재시작하지 않는다.

1. 롤백 직전의 기준값을 고정한다. 대상 플래그의 변경 번호, `pending`·`retry_wait`·`claimed` 행의 ID와 상태, 개별 소유권의 `owner_generation`, 활성 점유 토큰·점유 기한, 공급자 관련 기록을 함께 보관한다. 집계 건수만 저장하지 말고 반드시 대상 ID 목록을 남긴다.
2. `rollback_notification_dispatch_cutover_v1`을 정확한 예상 변경 번호와 새 요청 ID로 한 번만 호출한다. 기존 경로로 반환할 때는 그림자를 자동 복구하지 않도록 `p_reenable_shadow=false`를 유지한다. 변경 번호 충돌이나 범위 불일치가 나면 새 기준값을 임의로 만들어 재호출하지 말고 상태를 다시 읽은 뒤 인계를 중단한다.
3. 반환값과 실제 행을 대조한다. `rolledBack=true`, `mode`, `scopeKey`, `flagKeys`, `shadowEnabled=false`, 중단 래치 상태가 요청한 롤백과 정확히 일치해야 한다. `canceledCount`는 해당 범위에서 `pending`·`retry_wait`에서 롤백 취소로 바뀐 ID 수와, `cancelRequestedCount`는 `claimed`에 취소 요청이 기록된 ID 수와 정확히 같아야 한다. 대상 발송 플래그의 변경 번호는 직전보다 1 증가해야 하며, 그림자·설정 플래그는 실제 상태 변경 횟수를 반영한 기대 변경 번호와 같아야 한다. `flagRevisions`의 모든 값은 레지스트리 재조회 결과와 일치해야 하고, 영향을 받지 않은 플래그의 변경 번호는 변하지 않아야 한다. 변경 번호나 건수가 하나라도 다르면 소유권을 옮기지 않고 즉시 중단한다.
4. `cancelRequestedCount`에 포함된 모든 `claimed`의 점유와 점유 기한이 종료될 때까지 대기한다. 대기 중에는 업무 잠금을 잡지 않고, 작업 처리기가 취소 요청을 처리했거나 점유 기한 회수가 완료됐는지 재조회한다. 단순히 점유 기한 만료 시각이 지났다는 사실만으로 종료를 판정하지 않는다. 활성 점유 토큰·점유 기한이 없고 발송 상태가 `claimed`·`sending`이 아닌 것을 확인한 뒤에만 다음 단계로 간다. `cancelRequestedCount=0`이어도 대상 범위를 한 번 다시 조회해 활성 점유가 없는지 확인한다.
5. 종료 대기 후 개별 소유권을 다시 분류한다. `state=reserved`이고 `dispatch_started_at`, `dispatch_token`, `provider_reference`, `terminal_outcome`가 모두 없는, 즉 실제 전송을 한 번도 시작하지 않은 항목만 반환 후보이다. 발송 상태가 `sending`, `sent`, `delivery_unknown`이거나 소유권이 `closed`인 항목은 거부한다. 공급자 참조값, 메시지 ID, 응답 코드, 외부 시도 감사 기록 중 하나라도 있는 공급자 응답·영수증 보유 항목도 거부한다. 필드 간 상태가 서로 다르거나 전달 결과가 불명하면 임의로 보정하지 말고 보존·조사 대상으로 남긴다.
6. 반환 후보만 `transfer_notification_dispatch_ownership_v1`로 한 건씩 옮긴다. 전역 범위 소유자가 이미 기존 경로이고 해당 발송 플래그가 꺼진 상태에서, 저장해 둔 현재 `owner_generation`을 `p_expected_owner_generation`으로, `p_to_owner_kind='legacy'`를 대상으로, 건별 새 요청 ID와 승인된 사유 코드를 사용한다.
7. 소유권 반환값을 건별로 검증한다. `claim_id`가 요청한 ID와 같고, `owner_kind=legacy`, `state=reserved`, `reason_code`가 요청과 같아야 한다. 반환된 `owner_generation`은 직전 값보다 정확히 1 큰 10진 문자열이어야 한다. `ownership_transferred_pre_dispatch` 감사 기록의 이전·이후 소유자, 세대, 범위도 같은 값인지 확인하고, 이 반환 세대를 다음 단계의 유일한 기준값으로 저장한다.
8. 기존 경로는 `begin_legacy_notification_dispatch_v1`의 `p_expected_owner_generation`에 7번 단계에서 반환받은 `owner_generation`을 그대로 넣어 재시작한다. 수신자 세대인 `targetGeneration`을 넣거나 0·추정값·직전 세대를 대신 넣지 않는다. 반환값의 `acquired=true`, `claim_id`, `owner_generation`, `status=dispatch_started`가 모두 예상과 같을 때만 실제 공급자 호출을 시작한다. 오래된 세대·소유권 충돌이 발생하면 세대를 추측해 재시도하지 않고 해당 건을 중단한다.
9. 인계 완료 후에 마지막으로 범위별 건수와 변경 번호를 다시 대조한다. 반환된 건수와 거부·보존된 건수의 합이 최초 대상 ID 목록과 일치해야 한다. 부분 롤백이면 정상적인 다른 정규 소유자의 플래그·변경 번호·전송이 그대로인지, 반환된 범위에서 정규 외부 시도·부수 효과가 새로 발생하지 않았는지 확인한다. `sent`·`failed`·`delivery_unknown`, 공급자 응답·영수증, 설정 가져오기 기록, 감사 기록은 삭제·초기화되지 않아야 한다.

다음 중 하나라도 해당하면 소유권 반환과 기존 경로 재발송을 금지한다.

- 점유·점유 기한 종료를 재조회로 확인하지 못했다.
- 소유권이 `reserved`가 아니거나 전송 시작 흔적이 있다.
- 발송이 `sending`·`sent`·`delivery_unknown`이거나 공급자 응답·영수증이 있다.
- 롤백 변경 번호·취소 건수·인계 건수 중 하나라도 예상과 다르다.
- 소유권 반환 RPC가 돌려준 새 `owner_generation`으로 기존 경로 재시작을 검증하지 못했다.

코드 검증의 원자 롤백 연습은 운영 RPC를 호출하지 않는 순수 고정 검증 자료다. 부분 롤백, 전체 소유자 롤백과 그림자 복구, 알 수 없는 플래그·혼합 소유자, 오래된 예상 변경 번호를 모두 재현한다. 실패 고정 검증 자료에서는 플래그, 전달 행, 취소 요청, 소유권이 입력과 정확히 같아야 한다. 이 연습 결과는 운영 실행 승인이 아니며 실제 DB·플래그를 변경하지 않는다.

## 승인 경계

- 코드·미리보기 완료는 운영 그림자나 운영 전환 승인이 아니다.
- 7일 운영 그림자는 별도 승인 후에만 시작한다.
- 실제 소유자 전환은 순서별로 다시 승인받는다.
- 운영 플래그, 비밀정보, 실제 접속 주소, 공급자 본문은 코드 커밋에 넣지 않는다.
- Google Chat, Web Push, SOLAPI 실제 발송은 대상과 자격 증명이 확정된 별도 테스트 승인 없이는 통과로 기록하지 않는다.

## 보존

기존 설정·전송·공급자 상태 읽기는 정규 단독 운영을 시작한 뒤 최소 14일과 성공한 롤백 훈련 1회가 모두 끝날 때까지 유지한다. 삭제는 이 절차에 포함하지 않으며 별도의 검토 계획이 필요하다.
