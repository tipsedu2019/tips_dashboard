# 운영 안전성과 알림 완성 구현 계획 및 결과

이 문서는 등록·알림 업무의 구현 범위, 작업 0~22 진행 상태, 검증 증거를 한곳에서 확인하기 위한 한국어 기준 문서다. 코드 식별자와 명령어만 원문 표기를 유지한다.

작성 기준일: 2026-07-17

구현 브랜치: `codex/operational-safety-notification-completion`

작업 트리: `.worktrees/operational-safety-notification-completion`

## 1. 목표

- 등록 화면에 보이는 값과 실제 원자 저장 값이 항상 같게 만든다.
- 7개 업무의 알림 설정을 한 공통 제어면에서 영속 저장한다.
- 업무 이벤트, 대상 계산, 알림 의도, 전달, 읽음, 감사 기록을 재실행 안전하게 연결한다.
- 기존 발송 경로와 정규 발송 경로가 같은 발생 건을 동시에 보내지 못하게 한다.
- 그림자 검증에서는 실제 공급자 요청과 정규 알림함 투영이 모두 0건이어야 한다.
- 운영 전환과 롤백을 순서·변경 번호·소유권·점유 기한 기준으로 검증한다.
- 사용자가 확인하는 계획, 결과 보고, 운영 인계 문서는 한국어로 작성한다.

## 4. 작업별 구현 결과

### 작업 0: 안전한 기준선

상태: 완료

- 기존 작업 트리와 변경 내역을 보존한 별도 구현 작업 트리를 만들었다.
- 적용된 마이그레이션은 수정하지 않고 이후 시각의 순방향 마이그레이션만 추가했다.
- 기준선 테스트, 타입, 린트, 빌드, 브라우저 서버 소유권을 기록했다.
- 구현 중 Docker를 사용하지 않았다.

증거: `docs/operations/evidence/operational-safety-notification-baseline.md`

### 작업 1: 등록 생성·저장 진실성

상태: 완료

- 화면의 과목별 초기 진행값을 `createRegistrationCaseWithInitialWorkflow` 원자 RPC에 그대로 전달한다.
- 문의, 과목 트랙, 예약, 담당자, 이력이 한 요청으로 저장되거나 함께 롤백된다.
- 동일 논리 요청의 요청 지문과 요청 키를 고정해 재시도 중 중복 생성을 막는다.
- 전화상담은 예약 시각이 없는 과목별 대기 활동으로 유지한다.

### 작업 1A: 저장되지 않는 알림 설정 제거

상태: 완료

- 등록·전반·퇴원 화면에서 새로고침하면 사라지는 토글·템플릿을 저장된 설정처럼 보이지 않게 했다.
- 작업 1A 시점에는 기존 Google Chat 연결 관리와 발송 동작을 유지했다. 작업 20에서는 원문형 직접 발송을 계측 전용으로 폐쇄하고, 안정된 원본 이벤트 ID를 쓰는 고정 경로에 발송 책임을 한정했다.
- 공통 영속 설정 화면이 준비된 뒤 각 업무 진입점에서 같은 설정 UI를 사용하도록 연결했다.

### 작업 2: 등록 자동 이력 v2

상태: 완료

- 실제 등록 변이와 이력 이벤트를 같은 트랜잭션에서 기록한다.
- 행위자, 이전값, 이후값, 과목, 예약 ID, 변경 번호를 보존한다.
- v1 기록도 읽을 수 있는 호환 해석을 유지한다.

### 작업 3: 읽기 전용 등록 이력

상태: 완료

- 편집 필드를 이력처럼 보이던 영역에서 제거했다.
- 자동으로 기록된 주요 단계만 시간·행위자·과목 기준으로 보여 준다.
- 모바일과 데스크톱에서 동일한 의미와 순서를 유지한다.

### 작업 4: 공통 알림 어휘와 순수 계약

상태: 완료

- 7개 업무, 채널, 대상 그룹, 상태, 변경 번호, DTO 계약을 고정했다.
- PostgreSQL bigint는 JSON과 TypeScript 경계에서 10진 문자열로 처리한다.
- 불가능한 업무·채널·대상 조합은 저장 전에 거절한다.

### 작업 5: 비공개 정규 알림 스키마

상태: 완료

- 설정, 템플릿, 이벤트, 전달, 개인별 수신 영수증, 점유 기한, 감사, 요청 원장을 `dashboard_private`에 구성했다.
- 브라우저 역할의 정규 테이블 직접 접근을 닫고 고정 RPC만 공개했다.
- 기존 알림 이력과 읽음 값은 삭제하거나 초기화하지 않았다.
- 런타임 플래그 12개는 모두 `false`로 시작한다.

### 작업 6: 설정·감사·연결 API

상태: 완료

- 관리자·직원 읽기/쓰기 범위와 변경 번호 충돌을 검증한다.
- 한 번의 명시적 저장으로 규칙·템플릿·감사·요청 원장을 함께 반영한다.
- Google Chat 연결값은 서버에서 암호화하고 브라우저에는 설정 여부와 마스킹 정보만 돌려준다.
- 같은 요청 ID 재실행은 같은 결과를 반환하고 다른 내용 재사용은 거절한다.

### 작업 7: 내구성 작업 처리기

상태: 완료

- 생산, 점유, 점유 기한, 대상별 분배, 대상 재계산, 전달 적용, 알림함 반영을 고정 RPC로 구성했다.
- 공급자 호출 직전 원본·규칙·대상·소유권을 다시 검증한다.
- 전송 시작 뒤 결과가 불명확하면 `delivery_unknown`으로 보존하고 자동 재전송하지 않는다.
- 중복 외부 시도와 오래된 점유를 차단한다.

### 작업 8: 7개 업무 공통 설정 UI

상태: 완료

- `/admin/settings/notifications`에서 7개 업무 설정을 저장한다.
- 각 업무 화면의 설정 진입점도 같은 공통 패널을 사용한다.
- 저장 전 미저장 변경 상태, 충돌, 덮어쓰기 확인, 연결 설정을 분리했다.
- 설정 UI 플래그가 꺼져 있으면 기존 동작을 유지한다.

### 작업 9: 개인별 읽음과 푸시 알림 준비 상태

상태: 완료

- 공유 행의 `read_at`을 덮어쓰지 않고 사용자별 수신 영수증으로 읽음을 기록한다.
- 항목 이동과 읽음 처리를 분리해 같은 팀의 다른 사용자 상태를 보존한다.
- 서비스워커, 공개키, 서버키, 구독 소유자, 브라우저 권한, 자체 시험 상태를 구분한다.
- 준비되지 않은 상태를 성공처럼 표시하지 않는다.

### 작업 10: 등록 예약 달력

상태: 완료

- 목록·달력 전환, 월·주 보기, 예약 상태, 과목 표시를 구현했다.
- 예약 ID를 딥 링크의 유일한 식별자로 사용한다.
- 같은 날짜·종류의 여러 예약도 충돌하지 않는다.
- 달력에서 연 상세가 기존 등록 상세와 같은 원본을 사용한다.

### 작업 11: 등록 예약 리마인더 생산자

상태: 완료

- 전일 14:00, 당일 14:00, 1시간 전 규칙을 Asia/Seoul 기준으로 계산한다.
- 예약·규칙 변경 번호가 바뀌면 오래된 예약 알림을 취소한다.
- 예약 변경과 생산자 이력을 같은 트랜잭션으로 처리한다.

### 작업 12: 예약 편집·충돌 보존

상태: 완료

- 변경 번호 충돌이 나도 사용자가 입력한 초안을 잃지 않는다.
- 변경 전후 알림 예정값을 확인할 수 있다.
- 동일 작업의 처리 상태와 안전한 재시도 경계를 제공한다.

### 작업 13: 등록 알림 어댑터

상태: 완료

- 등록 공통, 전화상담, 방문상담, SOLAPI 고객 알림을 서로 다른 소유권 범위로 유지한다.
- A→B→A 수신자 변경과 이전 작업 대체를 검증한다.
- 방문상담 대상은 서버의 예약 참여자·담당자·변경 번호에서 계산한다.

### 작업 14: 7개 업무 어댑터

상태: 완료

- 7개 업무에 정확히 한 어댑터만 등록한다.
- 등록 어댑터만 예약 규칙과 예약 대상 재계산을 담당한다.
- 나머지 업무는 권위 이벤트 기반 즉시 알림만 처리한다.
- 작업 처리기 비밀값은 실행 시간 차이로 값을 추측할 수 없는 방식으로 비교한 뒤에만 점유를 시작한다.

### 작업 15: 할 일·영어 단어 재시험 생산자

상태: 완료

- 생성·수정·상태·댓글을 고정 원자 RPC로 전환했다.
- 댓글과 활동 이벤트의 ID 계약을 분리해 원본 ID를 정확히 해석한다.
- 단어 재시험 재시도는 기존 건 종료·연결·새 건 생성을 한 트랜잭션에서 처리한다.
- 브라우저의 사후 이벤트 쓰기와 두 번 호출 재시도를 제거했다.

### 작업 16: 전반·퇴원 생산자

상태: 완료

- 제출·완료 이벤트만 관리팀 알림 의도를 만든다.
- 체크리스트 저장은 종결 이벤트를 만들지 않는다.
- 명단 전환 완료와 감사·알림 원본을 한 트랜잭션에서 기록한다.
- 안정된 원본 이벤트 ID를 통해 기존 부수 효과와 정규 소유권을 연결한다.

### 작업 17: 휴보강 알림 상태

상태: 완료

- 기존 설정·템플릿·이력·발생 중복 방지 값을 보존해 가져온다.
- 승인과 재처리를 고정 서버 경계로 이동했다.
- 강의실이 겹치는 동시 승인 요청은 같은 강의실별 잠금을 고정 순서로 획득한 뒤 다시 확인해 둘 다 승인되는 경쟁 상태를 막았다.
- 휴보강 생성 응답이 유실된 재시도는 같은 요청 ID를 재사용해 같은 신청만 반환하고 중복 생성을 막는다.
- 기존 전달 실패가 휴보강 업무 저장 성공을 되돌리지 않도록 분리했다.
- 정규 그림자 전달과 기존 실제 전달을 같은 발생 건으로 중복 실행하지 않는다.

### 작업 18: 전자결재 생산자

상태: 완료

- 신청, 댓글, 결재자 변경, 승인·반려·철회 이벤트를 권위 변이와 함께 기록한다.
- 요청자, 현재 결재자, 관리팀 대상은 서버 원본에서 계산한다.
- 브라우저가 수신자나 이벤트 의미를 임의로 지정하지 못한다.

### 작업 19: 등록 인수인계

상태: 완료

- 등록 공통, 전화상담, 방문상담, SOLAPI의 의미와 실패 처리를 분리했다.
- 기존 경로 체크섬과 정규 경로 체크섬을 별도 보존해 실제 렌더링 차이를 숨기지 않는다.
- 전송 전 예약 소유권만 세대 일치 조건으로 기존 경로에 반환할 수 있다.
- 공급자 참조나 전송 시작 흔적이 있는 건은 재발송하지 않고 보존한다.

### 작업 20: 구형 계약 폐쇄와 운영 준비

상태: 구현 완료

- Google Chat·Web Push 원문형 구형 호출은 인증·무내용 계측 후 `skipped=true`로 끝나며 외부 공급자를 직접 호출하지 않는다.
- 안정된 원본 이벤트 ID는 서버의 고정 변환기와 고정 업무 경로만 사용한다.
- 고정 경로의 실제 본문 결과까지 확인해 성공·실패 영수증을 구분한다.
- 브리지 설치 뒤 최소 24시간의 최신 DB 증거가 없으면 폐쇄가 실패한다.

## 5. 운영 안전 계약

### 저장과 재실행

- 업무 데이터와 권위 이벤트는 같은 트랜잭션에서 성공하거나 함께 롤백한다.
- 요청 ID는 같은 내용 재실행만 허용한다.
- 예약 ID와 원본 이벤트 ID를 날짜나 표시 문구로 대체하지 않는다.
- PostgreSQL 큰 정수형 변경 번호와 세대는 10진 문자열 계약을 유지한다.

### 전달 소유권

- 하나의 발생 건은 기존 또는 정규 경로 중 정확히 한 소유자만 가진다.
- 외부 공급자 호출 직전에 소유권, 점유 토큰, 발송 토큰을 다시 확인한다.
- 같은 발송 토큰의 외부 시도는 한 번만 등록된다.
- 결과 불명은 자동 재시도하지 않는다.

### 그림자

- 정규 전달은 `skipped/shadow_mode`로 끝난다.
- 정규 공급자 요청과 정규 알림함 투영은 각각 0건이어야 한다.
- 운영의 10개 범위 증거는 실제 정규·기존 의도 일치와 현재 규칙 변경 번호·활성 템플릿 체크섬에 연결한다.
- 운영 전환 완전성은 범위당 임의 한 건이 아니라 모든 활성 규칙의 현재 자연 발생 비교로만 판정한다. 결정적 결과는 자연 발생 운영 비교를 대신할 수 없으며, 활성 규칙이 0개인 범위만 현재 상태에 묶인 별도 무활성 증거를 사용한다.
- 저빈도 경로의 결정적 검증 주기는 순수 로컬 코드·미리보기 추가 증거다. 정규 경로는 실제 어댑터와 공통 렌더러를 실행하고, 기존 경로는 주입 기록기와 독립 계산을 사용하며 네트워크·DB·공급자·알림함 작업이 모두 0건이어야 한다.
- 신규 템플릿 체크섬은 DB가 SHA-256으로 강제하고, 기존 32자리 체크섬은 과거 전달·감사 지문을 변경하지 않도록 자연 비교와 로컬 실행기에서 호환한다.
- 결정적 고정 검증 자료는 할 일·결재 댓글의 실제 원본 종류, 원본 이벤트와 업무 대상의 서로 다른 ID, 단일값·배열 중복을 포함한 다중 수신자, 휴보강 `subject_team`의 영어·수학 Google Chat 분기를 모두 검증한다.
- 서비스 역할이 제출한 결정적 배열·요약값·0건 숫자로는 운영 전환 증거를 만들 수 없다.
- 첫 전환 뒤 아직 기존 소유자인 범위의 설정이 기준값과 달라지면 해당 전환을 거절한다.

### 롤백

- `pending`·`retry_wait`는 취소한다.
- 전송 전 `claimed`에는 취소 요청을 남기고 점유·점유 기한 종료를 재조회한다.
- `sending`·`sent`·`failed`·`delivery_unknown`과 공급자 영수증은 보존한다.
- 기존 경로 반환은 `reserved`, 미전송, 공급자 참조 없음, 정확한 소유자 세대 조건을 모두 만족해야 한다.
