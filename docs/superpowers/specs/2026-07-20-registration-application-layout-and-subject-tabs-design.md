# 등록 신청서 양식 통일 및 과목 탭 설계

**작성일:** 2026-07-20
**상태:** 승인
**관련 설계:** `2026-07-19-registration-common-application-design.md`

## 1. 변경 목적

`등록 추가`와 저장된 신청서의 `상세보기·수정`이 같은 등록 문서를 다루면서도 문의 양식의 순서, 입력 방식, 과목 선택 표현이 서로 다르다. 저장된 신청서에서는 영어·수학 내용이 한 페이지에 연속해서 노출되고 자동 이력까지 본문을 길게 차지해, 사용자가 지금 어느 과목을 처리하는지 빠르게 파악하기 어렵다.

이번 변경은 다음 네 가지를 해결한다.

1. 신규 등록과 저장된 신청서의 문의 양식을 하나의 시각·상호작용 계약으로 통일한다.
2. 학년을 기준으로 환경설정에 등록된 같은 학교급의 학교만 선택하게 한다.
3. 과목별로 진행되는 모든 영역을 영어·수학 탭으로 전환한다.
4. 자동 이력을 본문에서 제거하고 헤더의 시계 아이콘으로 연다.

신청 건 하나가 문서 하나이고 영어·수학 트랙은 같은 문서 안에서 독립적으로 진행한다는 기존 원칙은 유지한다. 이번 문서는 기존 설계의 `6.1 문서 머리말`, `6.2 문의 정보`, `6.7 자동 이력` 및 본문에 모든 과목 트랙을 동시에 펼치는 표현 계약을 대체한다. 단계별 탭은 만들지 않으며, 새로 추가하는 탭은 과목을 선택하는 용도로만 사용한다.

## 2. 확정된 사용자 결정

- 등록 추가와 등록 수정·상세보기의 양식을 통일한다.
- 문의일시는 학생명 오른쪽에 둔다.
- 필수 항목은 가능한 한 왼쪽, 선택 항목은 오른쪽에 둔다.
- 문의 과목은 문의 정보 섹션의 가장 위에 둔다.
- 과목 선택의 활성·비활성 상태를 색과 체크 표시로 분명하게 구분한다.
- 과목별로 진행되어야 하는 영역 전체를 탭으로 전환한다.
- 자동 이력은 본문에 노출하지 않고 닫기 아이콘 왼쪽의 시계 아이콘으로 연다.

## 3. 검토한 접근

### 접근 A — 공용 표현 컴포넌트와 생성·상세 어댑터

문의 필드와 과목 선택 UI만 공용화하고, 신규 생성과 상세 수정의 저장·충돌·잠금 상태는 각자의 기존 로직이 담당한다.

- 장점: 두 화면이 다시 어긋나는 것을 막으면서 현재의 원자적 생성, 상세 충돌 복구, 섹션별 저장 계약을 보존한다.
- 단점: 값과 이벤트를 공용 표현 컴포넌트에 맞추는 얇은 어댑터가 필요하다.
- 결정: **채택**

### 접근 B — 신규와 상세의 전체 편집 상태 통합

생성·상세의 데이터 상태와 저장 로직까지 하나의 범용 편집기로 합친다.

- 장점: 코드 표면이 하나로 보인다.
- 단점: 첫 저장, 기존 문서 충돌 복구, 신원 잠금, 과목 동기화의 서로 다른 의미가 섞여 회귀 위험과 변경 범위가 크다.
- 결정: 채택하지 않는다.

### 접근 C — 두 화면의 JSX만 같은 모양으로 복제

각 화면의 상태 로직과 마크업을 그대로 둔 채 배치와 색상만 맞춘다.

- 장점: 최초 변경량이 작다.
- 단점: 다음 필드 변경 때 다시 두 화면이 갈라지고 학교 필터·접근성 계약도 중복된다.
- 결정: 채택하지 않는다.

## 4. 문의 정보 구조

문의 정보는 신규와 상세에서 아래 순서를 동일하게 사용한다.

### 4.1 과목 선택

문의 정보 섹션의 첫 번째 입력 묶음은 `문의 과목`이다.

- 영어와 수학은 체크박스 모양이 아니라 누를 수 있는 토글 버튼으로 표시한다.
- 선택된 과목은 기본 강조색 배경, 높은 대비의 글자색, 체크 아이콘을 함께 사용한다.
- 선택되지 않은 과목은 중립 배경의 테두리 버튼으로 표시한다.
- 색상만으로 상태를 전달하지 않고 `aria-pressed`와 체크 아이콘을 함께 제공한다.
- 상세 화면의 과목 동기화 저장은 기존 독립 저장 계약을 유지한다. 공통 정보 저장에 섞지 않는다.
- 진행 이력이 있는 과목을 제거할 수 없는 기존 제약과 안내를 유지한다.

### 4.2 공통 필드 배치

데스크톱에서는 다음 2열 배치를 사용한다.

| 행 | 왼쪽 | 오른쪽 |
| --- | --- | --- |
| 1 | 학생명 · 필수 | 문의일시 · 자동 |
| 2 | 학년 · 필수 | 학교 · 선택 |
| 3 | 학부모 전화 · 필수 | 학생 전화 · 선택 |
| 4 | 요청 사항 · 선택, 전체 너비 | — |

- 문의일시는 최초 생성 시 서버가 기록한 값을 읽기 전용 텍스트로 표시한다. 일반 입력창처럼 보이게 하거나 사용자가 변경하게 하지 않는다.
- 신규 작성 중 서버 문의일시가 아직 없으면 `저장 시 자동 기록`으로 표시한다.
- 필드의 `필수`, `선택`, `자동` 표시는 두 화면에서 같은 문구와 위치를 사용한다.
- 모바일에서는 각 행을 위에서 아래로 한 열로 쌓되 표의 읽기 순서를 유지한다.
- 기존 신원 잠금 규칙이 적용되는 상세 화면에서는 같은 필드 모양을 유지하고 해당 입력만 비활성화한다.

### 4.3 저장 경계

보이는 문의 양식은 같지만 저장 의미는 합치지 않는다.

- 신규 화면은 기존 원자적 등록 생성 계약으로 신청 건, 과목 트랙, 최초 진행 경로를 함께 만든다.
- 상세 화면의 공통 정보는 기존 버전 충돌 감지와 복구를 유지한 채 섹션 단위로 저장한다.
- 상세 화면의 과목 구성은 별도 과목 동기화 저장을 사용한다.
- 저장 실패·충돌·권한 제한 안내는 각 화면의 현재 권위 로직을 유지한다.

## 5. 학년별 학교 선택

학교 후보의 원천은 환경설정에서 관리하는 `academic_schools`다. 등록 저장값은 현재와 같이 학교명 문자열을 사용하므로 데이터베이스 저장 스키마는 바꾸지 않는다.

### 5.1 학교급 매핑

- `초1`~`초6` → `elementary`
- `중1`~`중3` → `middle`
- `고1`~`고3` → `high`

환경설정에 남아 있는 기존 별칭도 각각 같은 학교급으로 정규화한다.

- 초등: `elementary`, `elem`, `primary`, `초등`
- 중등: `middle`, `mid`, `secondary`, `중등`
- 고등: `high`, `highschool`, `고등`

### 5.2 후보 동작

- 학년을 선택하기 전에는 학교 선택을 비활성화하고 `학년을 먼저 선택`이라고 안내한다.
- 학년 선택 후에는 같은 학교급으로 설정된 학교만 후보로 표시한다.
- 후보는 환경설정의 정렬 순서, 학교명 순으로 안정 정렬한다.
- 학교는 선택 항목이므로 빈 선택을 항상 허용한다.
- 사용자가 학년을 다른 학교급으로 변경하면 새 후보에 없는 학교 초안은 초기화한다.
- 상세 화면을 처음 열었을 때 저장된 학교가 환경설정에서 삭제됐거나 현재 학년과 맞지 않아도 자동으로 지우지 않는다. `기존 입력` 후보로 보존하고, 사용자가 학년을 변경할 때만 정리한다.
- 알 수 없는 학년·학교 분류를 임의로 추정하지 않는다. 기존 값은 보존하되 신규 후보에는 포함하지 않는다.
- 학교 목록 조회가 실패하면 기존 저장값과 빈 값은 보존하고 신규 선택만 잠근다. 학교가 선택 항목이라는 이유로 등록 신청서 전체를 사용할 수 없게 만들지 않는다.

신규와 상세는 동일한 학년 옵션, 학교급 정규화 함수, 학교 후보 함수를 사용해야 한다.

## 6. 과목 탭

### 6.1 적용 범위

문의 정보 아래에 하나의 과목 탭 목록을 두고, 선택된 과목이 이후의 모든 과목별 진행 영역을 제어한다.

- 레벨테스트
- 상담
- 등록·대기 정보
- 입학 처리 안의 과목별 상태·수강 내용
- 과목 확인 필요, 문의 완료 재개, 미등록 완료 재개 같은 과목별 예외 처리

영어 탭에서는 영어에 해당하는 위 내용을, 수학 탭에서는 수학에 해당하는 내용을 보여 준다. 과목마다 단계가 달라도 같은 탭 선택이 모든 후속 섹션에 일관되게 적용된다.

### 6.2 탭 밖에 유지하는 공통 작업

다음은 신청 건 또는 공유 엔터티 단위이므로 과목 패널 안에서 복제하지 않는다.

- 여러 과목이 참여하는 공동 예약과 참여 과목 배지
- 신청 건 단위 입학신청서 발송 및 불확실 발송 복구
- 여러 과목의 수강 행을 묶는 입학 처리 배치
- 공통 신청 정보 저장

공동 예약이 선택된 과목을 포함하면 해당 섹션에 같은 예약 한 건을 보여 준다. 두 탭에서 보이더라도 동일한 `appointmentId`를 가리키며 별도 예약처럼 복제하지 않는다. 신청 건 단위 행동은 탭 선택과 무관하게 항상 도달할 수 있어야 한다.

### 6.3 상태와 편집 보존

- 탭은 `tablist`, `tab`, `tabpanel` 의미를 사용하고 키보드 화살표 이동과 포커스 상태를 제공한다.
- 선택 탭은 강조색과 명시적인 선택 상태를 함께 표시한다.
- 비활성 과목 패널은 React 트리에서 제거하지 않고 숨긴 상태로 유지한다. 탭을 바꿔도 미저장 초안과 dirty guard가 사라지면 안 된다.
- 섹션의 `진행 중`, `사용 가능`, `잠김` 상태 표시는 전체 과목의 OR 집계가 아니라 현재 선택 과목과 신청 건 단위 행동을 기준으로 계산한다.
- 일반적인 탭 클릭은 페이지를 자동 스크롤하지 않는다.
- 목록·달력의 깊은 링크로 처음 열 때만 대상 과목과 섹션에 최초 초점을 줄 수 있다.
- 과목 제거로 선택 중인 트랙이 사라지면 남은 첫 과목으로 안전하게 전환한다.

## 7. 자동 이력 진입

자동 이력은 계속 서버 기록을 사용하는 읽기 전용 기능이지만 공통 신청서 본문 섹션에서는 제거한다.

### 7.1 헤더 행동

- 저장된 신청서 헤더 오른쪽에서 닫기 아이콘 바로 왼쪽에 시계 아이콘을 둔다.
- 버튼의 접근 가능한 이름은 `자동 이력 보기`다.
- 신규 작성 화면에는 아직 이력이 없으므로 시계 아이콘을 표시하지 않는다.
- 기존의 과목·단계 필터와 상세 펼치기를 유지한다.
- 이력에서 추가·수정·삭제 행동은 제공하지 않는다.

### 7.2 표시 방식

시계 아이콘을 누르면 헤더에 붙은 별도 이력 패널을 연다.

- 데스크톱에서는 오른쪽 정렬의 넓은 팝오버 패널로 표시한다.
- 화면 높이를 넘으면 패널 내부만 스크롤한다.
- 작은 화면에서는 뷰포트 여백을 제외한 가용 너비를 사용한다.
- Escape, 바깥 클릭, 닫기 후 시계 버튼으로 포커스 복귀를 지원한다.
- 신청서 본문 스크롤 위치와 작성 중인 과목별 초안은 유지한다.

등록 신청서 다이얼로그가 공용 Sheet보다 높은 레이어를 사용하므로, 현재 구현에서는 다이얼로그 위 레이어가 보장된 공용 Popover를 우선 사용한다. Sheet를 사용하려면 먼저 공용 레이어 계약을 확장하고 별도로 검증해야 한다.

## 8. 변경 후 신청서 구조

저장된 신청서는 아래 구조를 사용한다.

1. 헤더: 학생명, 과목별 현재 상태, 자동 이력, 닫기
2. 문의 정보: 과목 선택, 공통 문의 필드, 저장 행동
3. 과목 탭: 영어·수학
4. 레벨테스트: 선택 과목 내용과 관련 공유 예약
5. 상담: 선택 과목 내용과 관련 공유 예약
6. 등록·대기 정보: 선택 과목 내용과 신청 건 공통 참조
7. 입학 처리: 선택 과목 내용과 신청 건 단위 행동

신규 신청서는 같은 헤더·문의 필드·후속 섹션 구조를 사용한다. 다만 자동 이력 버튼은 없고, 아직 생성되지 않은 과목별 후속 영역은 기존 선행 조건에 따라 잠긴 빈 상태로 표시한다.

## 9. 데이터와 오류 처리

- 등록 옵션 로더에 `academic_schools(id,name,category,sort_order)` 조회를 추가한다.
- 학교 카탈로그 상태는 담당자·수업 등 필수 카탈로그의 준비 상태와 분리한다.
- 학교 조회 실패가 등록 저장이나 기존 학교명 보존을 막지 않게 한다.
- 탭 선택은 화면 초점 상태이며 과목의 업무 상태나 데이터 값을 변경하지 않는다.
- 과목 탭 선택은 기존 깊은 링크의 `trackId`와 동기화할 수 있지만, 일반 탭 클릭 때 단계별 자동 스크롤은 실행하지 않는다.
- 알림, 예약, 입학 처리 RPC와 이력 생성 계약은 변경하지 않는다.

## 10. 검증 계약

### 10.1 단위·계약 테스트

- 공식 학년 12개와 학교급 매핑을 검증한다.
- 학교 분류의 표준값과 기존 별칭 정규화를 검증한다.
- 학년별 후보 제한, 정렬, 빈 이름 제외, 기존 저장값 보존을 검증한다.
- 등록 옵션 로더의 학교 조회와 독립 실패 처리를 검증한다.
- 신규와 상세가 동일한 과목 선택·문의 필드 컴포넌트를 사용하는지 검증한다.
- 정확한 행 순서와 필수·선택·자동 표시를 검증한다.
- 과목 탭의 의미 구조, 선택 상태, 비활성 패널 유지 계약을 검증한다.
- 자동 이력이 본문에서 제거되고 헤더 시계 버튼으로 열리는지 검증한다.

### 10.2 실제 브라우저 검증

- 신규와 저장된 신청서를 나란히 확인해 문의 양식 순서와 컨트롤이 같은지 검증한다.
- 학년별로 다른 학교급 후보가 섞이지 않는지 검증한다.
- 영어·수학 탭을 바꾸면 모든 과목별 섹션이 함께 전환되는지 검증한다.
- 한 탭에서 작성한 미저장 초안이 다른 탭을 다녀온 뒤 유지되는지 검증한다.
- 공동 예약과 신청 건 단위 입학 행동이 중복되거나 사라지지 않는지 검증한다.
- 시계 아이콘 위치, 이력 패널 스크롤, 닫기와 포커스 복귀를 데스크톱·모바일에서 검증한다.
- 공급자 알림은 비활성 상태를 유지하고 실제 메시지를 발송하지 않는다.

## 11. 완료 조건

- 신규와 상세의 문의 양식이 같은 공용 표현 계약을 사용한다.
- 문의 과목이 문의 정보 최상단에 있고 활성 상태가 색·아이콘·접근성 상태로 명확하다.
- 필드가 승인된 네 행 순서로 배치된다.
- 학교 후보가 선택한 학년의 학교급으로 제한된다.
- 과목별로 진행되는 전체 영역이 한 과목 탭 선택에 맞춰 전환된다.
- 탭 전환으로 미저장 내용이나 dirty guard가 사라지지 않는다.
- 자동 이력이 본문에 없고 헤더 시계 아이콘에서 읽기 전용으로 열린다.
- 기존 생성, 충돌 복구, 예약, 등록·입학, 이력, 무발송 안전 계약이 유지된다.
