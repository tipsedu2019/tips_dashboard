# 등록 업무 흐름 차트 및 로딩 최적화 설계

## 목표

등록 한 건이 문의부터 완료까지 이동하는 운영 기준을 상단 차트로 고정하고, 등록 화면의 초기 데이터 로딩을 실제 사용량에 맞게 줄인다. 상담 예약 알림과 레벨테스트 결과 링크는 차트 설명에만 남지 않고 현재 데이터 모델 안에서 실제로 동작해야 한다.

## 현재 문제와 근거

- 등록 화면은 68개 수업의 `schedule_plan` 약 5.29MB를 사용하지 않으면서도 매번 읽는다. 등록에 필요한 수업 필드는 약 13.4KB다.
- Next.js 개발 모드의 중복 effect가 동일한 데이터 로더를 동시에 호출하지만, 캐시는 완료된 결과만 보관해 두 요청을 합치지 못한다.
- 상단의 `상담/레벨테스트` 탭은 서로 다른 업무를 한 목록에 섞는다.
- 등록 운영 기준을 설명하는 차트가 없으며, 상담 예약 알림은 현재 일반 `processing` 알림에 섞여 실제 원장 개인 알림과 관리팀 Google Chat 발송을 보장하지 않는다.
- `level_test_material_link` 필드는 이미 존재하지만 이름이 Drive 링크 중심이라 시험지·결과지 URL이라는 운영 목적이 드러나지 않는다.

## 상단 구조

상단 순서는 다음과 같다.

1. 목록 탭: `문의 → 레벨테스트 → 상담 → 대기 → 등록 → 완료`
2. `등록 업무 흐름 · 운영 기준` 차트
3. 선택 탭의 세부 상태 필터
4. 목록 필터와 데이터 테이블

탭은 실제 목록 이동 수단이고 차트는 운영 기준이다. 차트의 접힌 상태에서도 6단계 순서를 보이며, 펼친 상태에서는 각 단계의 입력, 전이, 자동화, 완료 조건을 모두 보여 준다. 모바일에서는 내부 가로 스크롤 또는 세로 카드로 화면 전체의 가로 넘침을 만들지 않는다.

## 업무 흐름

1. **문의**: 기본 정보를 입력하고 관리팀이 레벨테스트, 성적 기반 바로 상담, 대기, 문의만 완료 중 다음 방향을 정한다. 고1 1학기 내신이나 모의고사 성적이 제출된 경우 레벨테스트를 생략할 수 있다.
2. **레벨테스트**: 예약, 진행, 결과 순서로 처리한다. 시험지와 결과지를 Google Chat 또는 Drive에 올린 뒤 `시험지·결과지 URL`을 기록한다.
3. **상담**: 예약, 진행, 결과 순서로 처리한다. 예약 시 프로필로 연결된 상담 책임자에게 대시보드 알림을 만들고 관리팀 Google Chat에 알린다. 상담 결과는 등록, 미등록, 대기 중 하나다.
4. **대기**: 현재 학기 수강반, 현재 학기 개강반, 다음 학기 개강반 중 하나로 구분한다. 등록 전환 전 레벨테스트 재응시 필요 여부를 확인하고 필요하면 레벨테스트 단계로 돌아간다.
5. **등록**: 입학 절차, 수납, MakeEdu 등록, 청구·교재·시간표 업무를 마친 뒤 완료한다.
6. **완료**: 등록 또는 미등록으로 닫는다. 문의만 종료는 미등록 계열의 별도 이력으로 유지한다.

현재 단일 링크/결과 필드는 한 번의 레벨테스트만 표현한다. 대기 후 재응시의 과거 시험지와 결과를 모두 보존하는 별도 응시 이력 테이블은 이번 변경에 포함하지 않는다. 차트에는 재응시 운영 규칙을 표시하되, 다회 응시 데이터 모델은 후속 설계 대상으로 남긴다.

## 탭과 상태 매핑

- `flow=inquiry`: `0.`
- `flow=level_test`: `1.`, `1-1.`
- `flow=consulting`: `2.`, `3.`
- `flow=waiting`: `4-1.`, `4-2.`, `4-3.`
- `flow=enrollment`: `5.`, `5-1.`, `6.`
- `flow=closed`: `7.`, `8.`, `9.`

기존 `flow=consulting` URL은 상담 탭으로 유지해 링크 호환성을 보존한다.

## 로딩 최적화

- 등록 화면의 수업 조회는 `id,name,subject,grade,teacher,room,textbook_ids`만 선택한다.
- `textbook_ids`가 없는 구형 스키마에서는 `id,name,subject,grade,teacher,room`으로 한 번만 폴백한다.
- 전반, 퇴원, 단어 재시험은 일정·수강생·대기자 데이터가 필요하므로 기존 전체 수업 조회를 유지한다.
- 완료 캐시와 같은 키로 진행 중 Promise도 보관한다. 동일 키의 동시 호출은 같은 Promise를 공유한다.
- 캐시 무효화 후 이전 Promise가 완료되어도 새 캐시를 덮어쓰지 않도록 Promise identity를 확인한다.

## 상담 예약 알림

- 등록에서 `ops_tasks.secondary_assignee_id`를 상담 책임자의 안정적인 프로필 연결로 사용하고, 기존 `registration.counselor`는 표시 이름으로 유지한다.
- 상담 책임자 입력은 자유 텍스트 대신 `profile_id`가 연결된 원장 선택값을 사용한다.
- `2. 상담 예약` 저장 시 예약일시와 상담 책임자가 모두 필요하다.
- 전용 서버 경로 `/api/registration/consultation-notification`은 사용자 입력 메시지를 그대로 중계하지 않고 현재 DB의 등록 건을 다시 읽는다.
- 서버는 `staff` 또는 `admin`만 허용하고, 등록 건·상담 예약 상태·수신 프로필을 검증한다.
- 원장 개인 대시보드 알림과 관리팀 Google Chat을 각각 예약일시 기반 dedupe key로 한 번만 발송한다.
- 알림 실패는 저장된 상담 예약을 되돌리지 않으며, 화면에 `예약은 저장됐지만 알림을 확인하세요` 경고를 표시한다.

## 테스트와 완료 기준

- 등록 클래스 조회에서 `schedule_plan`, `schedule`, `fee`, `student_ids`, `waitlist_ids`가 빠진다.
- 동일 키의 동시 로드가 한 번의 Supabase request wave만 만든다.
- 탭 순서와 상태 매핑이 6개로 고정된다.
- 차트는 탭 행 아래, 상태 필터와 테이블 위에 렌더되고 접근성 이름과 펼침 상태를 제공한다.
- 상담 예약 알림 경로는 인증, 역할, 등록 상태, 수신자, dedupe key를 검증한다.
- 실제 브라우저에서 첫 데이터 표시 시간, 데스크톱/모바일 차트, 레벨테스트/상담 탭, 콘솔 오류를 확인한다.

## 범위 제외

- 다회 레벨테스트 응시 이력 테이블
- MakeEdu API 자동 등록 또는 수납 연동
- 기존 전반·퇴원·단어 재시험의 수업 데이터 조회 축소
- 배포, 커밋, 운영 데이터 마이그레이션
