# 영어 단어 재시험 재재시험·업무 매뉴얼 설계

**작성일:** 2026-07-21
**상태:** 승인

## 1. 변경 목적

영어 단어 재시험에서 불합격 또는 미응시가 확정된 뒤 후속 시험을 만들 때, 기본 재시험과 후속 재재시험을 명확히 구분한다. 재재시험은 이전 본시험일을 기본값으로 이어받지만 본시험일 기준 자동 미응시 기한을 적용하지 않는다. 업무흐름 차트는 개별 업무 모달에서 제거하고 메인 데이터테이블의 전역 매뉴얼로 이동한다.

## 2. 용어와 식별 기준

- `기본 재시험`: 일반 추가로 만든 최초 단어 재시험이다. `retry_of_task_id`가 없다.
- `재재시험`: 불합격 또는 미응시 업무에서 `재재시험 추가`로 만든 후속 단어 재시험이다. `retry_of_task_id`가 있다.
- `재재시험 추가됨`: 원본 업무에 `retry_task_id`가 있어 후속 재재시험이 생성된 상태다.
- 새 분류 컬럼은 만들지 않는다. 기존 `retry_of_task_id`, `retry_task_id` 양방향 연결을 계보와 자동 미응시 기한 면제의 단일 기준으로 사용한다.
- 런타임 조회 모델에는 두 연결 ID를 읽기 전용 `retryOfTaskId`, `retryTaskId`로 노출한다. 일반 생성·수정 payload가 연결 ID를 직접 쓰는 것은 계속 금지한다.

## 3. 상태와 생성 흐름

### 3.1 불합격

1. 조교선생님이 점수를 저장하고 불합격을 보고한다.
2. 담당선생님은 `재재시험 추가` 또는 `불합격 확인`을 선택한다.
3. `재재시험 추가`를 선택하면 새 폼은 원본의 본시험일을 기본값으로 유지하며 수정할 수 있다.
4. 저장 버튼은 `재재시험 추가 및 불합격 확인`으로 표시한다.
5. 저장 성공 시 원본 업무를 완료하고 새 재재시험을 `요청/시작 전`으로 만들며 양방향 연결과 두 이력 이벤트를 한 트랜잭션에 기록한다.

### 3.2 미응시

1. 기본 재시험이 본시험일 기준 7일을 경과하면 자동으로 미응시 보고 대상이 된다.
2. 담당선생님은 `재재시험 추가` 또는 `미응시 확인`을 선택한다.
3. `재재시험 추가` 폼은 원본 본시험일을 기본값으로 유지하며 수정할 수 있다.
4. 저장 버튼은 `재재시험 추가 및 미응시 확인`으로 표시한다.
5. 저장 성공 시 원본 업무 상태만 완료한다. 원본 세부 결과 `미응시`는 보존하고 새 재재시험을 `요청/시작 전`으로 만든다.

### 3.3 이미 미응시 확인을 완료한 경우

1. 담당선생님이 먼저 `미응시 확인`을 눌러 원본 업무가 완료 목록으로 이동했더라도, 원본 결과가 `미응시`이고 아직 후속 링크가 없으면 `재재시험 추가`를 제공한다.
2. 저장 성공 시 원본의 완료 상태, 기존 완료 시각, `미응시` 결과와 완료 이력을 모두 보존한다.
3. 새 재재시험과 양방향 링크, `word_retest.retry_created` 이벤트만 추가하며 기존 `word_retest.completed` 이벤트를 중복 기록하지 않는다.
4. 같은 요청 ID 재실행은 최초 후속 업무와 한 개짜리 원본 이벤트 배열을 그대로 반환한다.

### 3.4 공통 제약

- 원본 하나에는 후속 재재시험 하나만 직접 연결할 수 있다.
- 후속 재재시험에서 다시 불합격 또는 미응시가 발생하면 그 업무를 원본으로 다음 재재시험을 만들 수 있다.
- 요청 멱등성과 기존 고유 연결 제약으로 중복 클릭이나 응답 유실 재시도에서 중복 업무 생성을 막는다.
- 기존 `word_retest.completed`, `word_retest.retry_created` 원본 이벤트와 양방향 링크를 유지한다.

## 4. 자동 미응시 규칙

- 기본 재시험만 `본시험일 + 7일` 자동 미응시 규칙을 적용한다.
- `retryOfTaskId`가 있는 모든 재재시험은 본시험일이 과거여도 자동 미응시로 전환하지 않는다.
- 화면의 자동 대상 계산과 서버의 `deadline` 미응시 RPC가 같은 조건을 적용한다.
- 서버는 재재시험에 대한 `deadline` 출처 요청을 거절한다. 화면 우회나 오래된 클라이언트에서도 기한 면제가 깨지지 않아야 한다.
- 사용자가 명시적으로 수행하는 수동·출석 기반 미응시 처리의 기존 권한과 규칙은 바꾸지 않는다.

## 5. 목록과 상세 표시

- 완료된 원본의 기존 결과 문구인 `미완료: 불합격` 또는 `미응시`는 유지한다.
- 원본에 `retryTaskId`가 있으면 결과 옆에 `재재시험 추가됨` 보조 배지를 표시한다.
- 후속 업무에 `retryOfTaskId`가 있으면 `재재시험` 보조 배지를 표시한다.
- 목록 행과 상세 헤더에서 같은 배지 컴포넌트와 문구를 사용한다.
- 결과 의미와 후속 조치 표시는 서로 대체하지 않는다.

## 6. 업무 매뉴얼과 플로우 차트

- 메인 데이터테이블 툴바에서 영어 단어 재시험 알림 아이콘 바로 왼쪽에 항상 보이는 매뉴얼 아이콘을 둔다.
- 아이콘의 접근 가능한 이름과 제목은 `영어 단어 재시험 업무 매뉴얼`이다.
- 아이콘을 누르면 전역 매뉴얼 모달에서 정적인 전체 업무흐름 차트를 보여 준다.
- 수정 모달과 상세 모달의 기존 `WordRetestProgressStepper`와 펼침형 업무흐름 차트는 제거한다. 상세 헤더의 현재 상태 배지는 유지한다.
- 매뉴얼은 알림 기능 플래그나 관리 권한과 관계없이 열 수 있어야 한다.

플로우 차트에는 다음 내용을 명시한다.

- 기본 재시험: `재시험(기본) 추가 → 시험 시작 → 점수 입력·저장 → 결과 판정`
- 기본 미응시: `본시험일 + 7일 → 미응시 보고 → 미응시 확인 또는 재재시험 추가`
- 불합격: `불합격 보고 → 불합격 확인 또는 재재시험 추가`
- 합격: `합격 보고 → 합격 확인`
- 재재시험: `이전 본시험일 기본 유지`, `자동 미응시 기한 없음`

## 7. 오류 처리와 저장 정합성

- 재재시험 생성은 원본 완료, 새 업무 생성, 양방향 링크, 이력 이벤트 기록을 한 서버 트랜잭션에서 처리한다.
- 이미 후속 업무가 있거나 원본 상태가 바뀐 경우 충돌 오류를 반환하고 최신 목록을 다시 불러오도록 안내한다.
- 미응시 원본의 `retest_status=absent`를 `done`으로 덮어쓰지 않는다.
- 저장 실패 시 원본 완료나 링크 일부만 남지 않아야 한다.

## 8. 마이그레이션과 배포 경계

- 적용된 기존 마이그레이션을 수정하는 대신 신규 마이그레이션에서 재시험 재생성 RPC와 deadline 미응시 검증을 갱신한다.
- 기존 알림 계약 폐쇄 안전 게이트는 수정하거나 우회하지 않는다.
- 운영 반영 전 앱 코드와 신규 DB 함수가 함께 준비됐는지 확인한다. DB가 준비되지 않은 상태에서 미응시 재재시험 액션만 먼저 노출하지 않는다.
- Google Chat, Web Push, SOLAPI 등 외부 알림 제공자 설정은 이번 변경 범위에서 활성화하지 않는다.

## 9. 검증 범위

- 불합격과 미응시에서 각각 재재시험을 생성할 수 있다.
- 새 폼에 이전 본시험일이 기본값으로 나타나고 사용자가 수정할 수 있다.
- 미응시 원본 결과가 보존되고 원본·후속 업무 연결이 정확하다.
- 재재시험은 과거 본시험일이어도 자동 미응시 대상이 아니다.
- 기본 재시험은 기존 7일 자동 미응시 규칙을 유지한다.
- 중복 생성과 오래된 상태 저장이 차단된다.
- `미응시 확인`으로 이미 완료된 원본에서도 후속 링크가 없으면 재재시험을 만들 수 있고, 원본 완료 시각과 기존 완료 이력은 바뀌지 않는다.
- 원본의 `재재시험 추가됨`, 후속의 `재재시험` 배지가 목록과 상세에서 일치한다.
- 매뉴얼 아이콘이 알림 아이콘 바로 왼쪽에 있고, 수정·상세 모달에는 기존 플로우 차트가 남지 않는다.
- 매뉴얼 차트가 기본 재시험과 재재시험의 기한 차이를 설명한다.
- 관련 모델·서비스·SQL 계약·워크스페이스 테스트, 전체 테스트, ESLint, TypeScript, Webpack 프로덕션 빌드가 통과한다.
- 실제 브라우저에서는 운영 데이터를 생성하지 않는 안전한 경로로 매뉴얼 위치, 문구, 배지, 날짜 기본값을 확인한다.

## 10. 범위 밖

- 재재시험 전용 별도 테이블이나 독립 메뉴를 만들지 않는다.
- 자동 미응시 외의 알림 발송 정책과 제공자 설정을 바꾸지 않는다.
- 기존 재시험·재재시험 기록을 일괄 변환하지 않는다. 이미 저장된 양방향 연결 정보만 표시와 기한 판단에 사용한다.

## 11. 조교선생님 권한 경계

- 실제 `assistant` 계정은 영어 단어 재시험의 조교 단계 공동 큐 전체를 조회할 수 있어야 한다. 개별 조교 프로필 ID나 누락된 팀 메타데이터 때문에 목록이 비어서는 안 된다.
- 조교 계정의 팀 메타데이터가 없으면 영어 단어 재시험 공동 큐에서만 `조교팀`으로 보정한다. 조교 화면은 `조교선생님` 탭에 고정하고 담당선생님 탭이나 `role=teacher` 직접 전환을 허용하지 않는다.
- 조교선생님은 재시험 접수 추가, 시험 시작, 점수 입력·저장, 미응시 처리, 합격·불합격 결과 보고까지 수행할 수 있다.
- 결과 보고 뒤의 담당선생님 단계인 합격·불합격·미응시 최종 확인과 재재시험 추가·수정 요청은 기존 담당선생님 권한으로 유지한다.
- 브라우저 액션 가시성만으로 권한을 구분하지 않는다. 기존 공개 `SECURITY DEFINER` RPC의 서명은 유지하면서 조교 요청의 일반 수정·상태 전이·수정 요청·재재시험 생성 경계를 서버에서 검사하고, 일반 할 일의 기존 조교 권한은 바꾸지 않는다.
- 조교선생님 화면에서는 `휴보강` 메뉴와 검색 결과를 노출하지 않으며 `/admin/makeup-requests` 직접 접근도 허용하지 않는다.
- 휴보강 데이터 조회와 변경은 화면 차단에만 의존하지 않는다. 제한 정책, 직접 변경 가드, 공개 휴보강 RPC와 서비스 역할 API에서도 `assistant` 역할을 거절한다.
- 조교선생님의 전역 알림함에서도 레거시와 canonical 휴보강 알림의 제목·본문·링크를 모두 숨기되, 일반 업무 알림은 그대로 유지한다.
- 관리자·관리팀·담당선생님의 기존 영어 단어 재시험 및 휴보강 권한은 유지한다.
