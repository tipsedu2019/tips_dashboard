# 등록 공통 신청서 설계

**작성일:** 2026-07-19
**상태:** 설계 방향 승인, 문서 검토 대기

## 1. 목표

등록 업무의 생성부터 완료까지 학생의 등록 신청 건 하나를 유일한 운영 문서로 사용한다.

운영자는 `문의 정보 → 레벨테스트 → 상담 → 등록·대기 정보 → 입학 처리 → 자동 이력`이 한 장에 놓인 같은 신청서를 계속 열고, 진행상태에 따라 빈 값을 채워 나간다. 목록의 상태 탭은 문서를 나누는 단계가 아니라 동일한 신청 건을 찾기 위한 보기 필터다.

이 설계는 현재 노션 `등록관리`의 한 페이지·누적 입력 방식과 전반·퇴원·휴보강의 공통 신청서 경험을 등록 업무의 복잡한 과목별 흐름에 맞게 완성한다.

## 2. 설계 원칙

1. **신청 건 하나가 문서 하나다.** 단계나 과목이 바뀌어도 새 문서를 만들지 않는다.
2. **빈 값도 문서의 일부다.** 미래 단계는 숨기지 않고 빈 칸으로 보여 주되, 선행 조건이 충족되기 전에는 입력을 잠근다.
3. **영어와 수학은 같은 문서 안에서 독립적으로 진행한다.** 공통 학생 정보는 한 번만 저장하고, 과목별 상태·담당자·일정·결과·수업은 서로 덮어쓰지 않는다.
4. **상태는 업무 결과다.** 사용자가 상태값을 직접 바꾸는 자유형 드롭다운을 두지 않는다. 예약, 결과 저장, 대기 전환, 등록 결정, 입학 처리 같은 명시적 행동이 기존 서버 규칙을 통해 상태를 변경한다.
5. **한 페이지는 한 번에 전부 저장한다는 뜻이 아니다.** 예약·상담·입학 처리의 원자성, 충돌 감지, 알림 후속 처리를 보존하기 위해 섹션별로 저장한다.
6. **이력은 자동 기록하고 수정하지 못한다.** 책임자와 처리 시각은 서버 이벤트로 남기고 신청서 하단에서 읽기 전용으로 보여 준다.
7. **필드 순서는 업무 문장을 따른다.** 새 입력 묶음은 실제로 존재하는 값을 `누가 → 언제 → 어디서 → 무엇을 → 어떻게` 순서로 배치한다. 해당 값이 없는 업무에 가짜 담당자·장소 필드를 만들지는 않는다. 처리자와 처리 시각은 편집 필드로 반복하지 않고 자동 이력에 남긴다. 상담 영역의 기준 순서는 `상담 책임자 → 방문상담일시 → 방문상담실`이다.

## 3. 용어와 문서 단위

- **등록 신청 건(case):** 한 학생의 한 번의 등록 문의에서 시작한 부모 업무다. 공통 학생·보호자·문의 정보를 소유한다.
- **과목 트랙(track):** 등록 신청 건 안의 영어 또는 수학 진행상태다. 두 과목은 서로 다른 단계에 있을 수 있다.
- **공통 신청서(application):** 부모 업무와 모든 과목 트랙, 예약, 시험, 상담, 대기, 수강, 입학 처리, 자동 이력을 한 화면에 조합한 사용자 문서다.
- **상태 보기(view):** `문의`, `레벨테스트`, `상담`, `대기`, `등록`, `완료` 목록 필터다. 별도 문서나 별도 데이터 소유자가 아니다.

## 4. 검토한 대안

### 대안 A — 공통 신청서 + 섹션별 저장

모든 섹션을 한 페이지에 유지하고 실제 업무 입력도 해당 섹션 안에 둔다. 저장은 기존 원자적 업무 단위별로 수행한다.

- 장점: 노션식 누적 문서 경험과 현재 서버 안전 규칙을 모두 만족한다.
- 단점: 현재 분리된 편집기를 섹션 안으로 재배치해야 한다.
- 결정: **채택**

### 대안 B — 공통 신청서 + 전체 저장 버튼 하나

페이지 전체 초안을 한 번에 전송한다.

- 장점: 일반 문서 편집과 가장 비슷해 보인다.
- 단점: 예약, 상담 결과, 입학 처리, 알림 후속 작업이 한 요청에 섞이고 동시 수정 충돌 범위가 커진다.
- 결정: 채택하지 않는다.

### 대안 C — 현재 요약 화면을 유지하고 편집기만 펼치기

현재 단계의 외부 편집기와 팝업을 유지하되 접힌 영역만 줄인다.

- 장점: 구현 범위가 가장 작다.
- 단점: 사용자는 계속 요약, 과목 선택, 별도 편집기, 팝업 사이를 오가야 하므로 한 문서를 채운다는 경험이 완성되지 않는다.
- 결정: 채택하지 않는다.

## 5. 목록 설계

### 5.1 신청 건 중심 행

목록의 한 행은 과목 트랙이 아니라 등록 신청 건 하나를 나타낸다.

- 학생명과 공통 문의 정보를 한 번만 표시한다.
- 영어·수학이 모두 있으면 같은 행에 과목별 현재 상태 배지를 표시한다.
- 행을 어느 탭에서 열어도 같은 공통 신청서를 연다.
- 현재 탭과 일치하는 과목을 신청서의 최초 초점으로 사용할 수 있지만, 별도 문서를 만들거나 다른 과목을 숨기지 않는다.
- 같은 탭에 두 과목이 함께 해당되면 현재 탭의 기존 과목별 정렬 규칙으로 두 트랙을 먼저 정렬하고, 그중 첫 트랙의 정렬값을 신청 건 행의 대표 정렬값으로 사용한다. 동률은 신청 건 식별자로 안정 정렬한다.
- 행 안에서는 현재 탭과 일치하는 각 과목의 담당자와 처리 시각을 각각 표시한다. 서로 다른 값을 하나의 신청 건 담당자·시각으로 합치지 않는다.
- 목록의 빠른 행동을 유지한다면 반드시 과목명을 붙인 과목별 행동으로 제공한다. 신청 건 전체에 적용되는 것처럼 보이는 단일 빠른 행동은 두지 않는다.

### 5.2 상태 탭의 포함 규칙

신청 건은 해당 단계에 속한 과목 트랙이 하나라도 있으면 그 상태 보기에 포함된다.

예를 들어 영어가 `상담`, 수학이 `등록`이면 같은 신청 건이 상담 보기와 등록 보기에 각각 한 번씩 나타난다. 각 보기의 건수는 그 보기에 포함된 **고유 신청 건 수**다. 따라서 탭별 건수를 합친 값은 전체 신청 건 수와 같을 필요가 없다.

위 포함 규칙의 예외로, 완료 보기는 모든 과목이 `등록 완료`, `미등록`, `문의 종료` 같은 종료 상태일 때만 포함한다. 한 과목이라도 진행 중이면 그 신청 건은 완료 보기에 들어가지 않는다. 따라서 종료된 한 과목의 결과는 진행 중인 신청서 안에서는 계속 보이지만 완료 목록에는 중복 노출하지 않는다.

### 5.3 달력 보기

달력은 현재처럼 구체적인 예약 엔터티를 기준으로 표시한다.

- 레벨테스트와 방문상담은 날짜가 아니라 `appointmentId`를 식별자로 사용한다.
- 같은 신청서의 영어·수학이 한 예약을 공유하면 달력 항목도 하나이며 과목 배지를 함께 표시한다.
- 달력 항목을 열면 같은 공통 신청서를 열고 해당 예약 섹션으로 이동한다.

## 6. 공통 신청서 구조

신청서는 내부 단계 탭이나 이전·다음 버튼 없이 아래 순서를 고정한다.

### 6.1 문서 머리말

- 학생명
- 영어·수학 과목별 현재 상태 배지
- 닫기

신청 건에는 과목과 별개인 단일 진행상태를 새로 만들지 않는다. `현재 업무` 보조 카드도 제거한다. 각 과목의 현재 단계는 머리말과 해당 섹션의 상태 배지에서 바로 알 수 있어야 한다.

### 6.2 문의 정보

- 과목 구성
- 학생명
- 학년
- 학교
- 학부모 전화
- 학생 전화
- 요청 사항
- 문의일시(읽기 전용)

현재 정규 모델에서 제거된 `문의 채널`은 노션에 존재하더라도 이 설계에서 다시 도입하지 않는다. 별도 요구와 저장 계약이 승인되기 전에는 화면만 있는 가짜 필드를 만들지 않는다.

학생·보호자·요청 정보는 과목의 현재 상태와 무관하게 권한과 기존 신원 잠금 규칙이 허용하는 범위에서 계속 수정할 수 있다. 과목 구성도 상태 매트릭스와 별개로 기존 과목 동기화 계약을 사용하며, 일반 공통 정보 저장에 섞지 않는다. 진행 이력이 있는 과목을 임의로 삭제할 수 없는 현재 제약도 유지한다. 문의일시는 자동으로 처음 생성된 값을 사용하며 일반 입력창에서 임의로 바꾸는 필드로 만들지 않는다.

### 6.3 레벨테스트

과목별로 아래 값을 같은 섹션에서 보여 준다.

- 진행상태(읽기 전용 배지)
- 예약일시
- 장소
- 시험 시작·완료 상태
- 시험지·결과지 링크
- 결과

공유 예약은 한 편집 영역에 참여 과목 배지를 표시한다. 예약 추가·변경·취소와 시험 시작·결과 저장은 이 섹션 안에서 수행한다. 예약 편집기를 신청서 바깥에 별도로 렌더링하지 않는다.

아직 레벨테스트 경로가 아닌 과목도 빈 필드를 볼 수 있지만 입력은 잠긴다. 잠금 사유는 `레벨테스트 진행을 선택하면 입력할 수 있습니다`처럼 한 줄로 표시한다.

### 6.4 상담

과목별로 아래 순서를 사용한다.

1. 상담 책임자
2. 전화상담 대기 기준일시
3. 방문상담일시
4. 방문상담실
5. 상담 결과

전화상담 대기 기준일시는 서버가 기록한 읽기 전용 값이다. 사용자가 예약 시각처럼 편집하지 않는다. 방문상담 예약과 상담 결과 저장도 같은 상담 섹션에서 처리한다.

영어와 수학의 상담 책임자는 서로 다를 수 있다. 담당자 자동 배정, 수동 재지정, 재배정이 필요한 방문상담 보정 규칙은 현재 서버 계약을 유지한다.

### 6.5 등록·대기 정보

과목별로 아래 값을 같은 섹션에 둔다.

- 대기 종류
- 대기 수업
- 등록 단계(읽기 전용 배지)
- 수강 수업
- 교재
- 수업 시작일·회차
- 입학 처리 시작 행동
- 문의 요청 사항(읽기 전용 참조)

진행상태나 등록 여부를 자유형 값으로 편집하지 않는다. `등록 결정`은 상담 결과 또는 대기 전환의 결과로 도달하는 읽기 전용 상태다. 수업과 시작 일정을 검증한 뒤 실행하는 별도 행동은 `입학 처리 시작`이다. 문의 요청 사항은 문의 정보와 같은 원본을 읽기 전용으로 참조하며 두 번째 편집 필드나 복제 값을 만들지 않는다. 여러 수업 등록은 현재처럼 반복 가능한 행으로 표시한다. 등록 대신 대기 또는 미등록으로 전환하는 행동도 이 섹션 안에 둔다. 과목별 값은 독립적으로 저장한다.

### 6.6 입학 처리

입학 처리 체크리스트는 신청 건 안에 저장된 수강 행을 대상으로 한다.

1. 입학신청서 발송
2. 메이크에듀 등록
3. 청구서 발송
4. 수납 완료 확인
5. 등록 완료

1단계 `입학신청서 발송`은 저장된 수강 행이 없어도 하나 이상의 과목이 `등록 결정` 상태가 되면 입력 가능하다. 이 메시지는 과목별이 아니라 신청 건 단위이므로 발송 행동은 신청서에 하나만 두고 대상 과목 배지를 함께 표시한다. 2–5단계는 대상 수강 행이 저장되고 입학 처리 배치가 시작되기 전까지 빈 상태로 보이되 잠긴다. 대상 수강 행을 선택한 뒤 기존 배치 단위 처리 규칙을 사용한다. 메시지 발송과 상태 동기화가 불확실할 때는 재전송하지 않고 기존 확인·조정 흐름을 노출한다.

### 6.7 자동 이력

자동 이력은 공통 신청서의 마지막 섹션이다.

- 시간 역순으로 주요 업무 이벤트를 표시한다.
- 행위자, 시각, 과목, 단계, 결과를 서버 기록에서 가져온다.
- 직접 추가·수정·삭제할 수 없다.
- 과목·단계 필터는 유지할 수 있지만 이력과 실제 업무 입력을 다른 문서처럼 분리하지 않는다.

## 7. 생성부터 완료까지 같은 문서 사용

### 7.1 신규 신청서

`등록 추가`는 상세와 같은 공통 신청서 셸을 연다.

- 모든 섹션이 처음부터 같은 순서로 보인다.
- 문의 정보와 과목 선택은 즉시 입력 가능하다.
- 과목별 다음 업무는 현재 계약과 같은 `문의 유지`, `바로 전화상담`, `레벨테스트`, `방문상담` 네 가지다.
- `문의 유지`는 이후 섹션을 잠긴 빈 상태로 둔다.
- `바로 전화상담`은 상담 책임자만 입력하게 하고 전화상담 대기 기준일시는 생성 시 서버가 기록한다.
- `레벨테스트`는 레벨테스트 예약일시·장소와 참여 과목을 입력하게 한다.
- `방문상담`은 상담 책임자·방문상담일시·방문상담실과 참여 과목을 입력하게 한다.
- 두 과목이 같은 예약 경로를 선택하면 하나의 공유 예약 입력 묶음과 과목 배지로 표시한다.
- 등록·대기와 입학 처리 같은 미래 섹션은 빈 상태로 보이되 잠겨 있다.

신규 원자적 생성에서만 `방문상담` 직행을 허용한다. `문의 유지`로 저장된 과목은 현재 서버 전이 계약대로 먼저 `바로 전화상담`을 통해 전화상담 대기에 들어간 뒤 방문상담을 예약한다.

첫 저장은 현재의 원자적 등록 생성 계약을 사용해 부모 신청 건, 과목 트랙, 선택한 초기 예약·상담 경로를 함께 만든다. 성공 후 창을 닫았다가 다른 상세 화면을 여는 대신, 같은 신청서 셸을 생성 모드에서 저장된 신청 건 모드로 다시 불러온다.

### 7.2 저장된 신청서

저장 후에도 섹션 순서와 화면 구조는 바뀌지 않는다. 진행상태는 다음 섹션의 입력 가능 여부와 현재 상태 강조만 바꾼다. 이전 단계 값과 미래 단계의 빈 필드는 계속 같은 자리에 남는다.

단계 후퇴, 예약 취소, 미등록 전환, 입학 처리 취소는 필드 값을 직접 지우거나 상태 배지를 편집해서 수행하지 않는다. 해당 섹션의 명시적 취소·전환 행동과 기존 이력 생성 RPC를 사용한다.

기존 예외 업무도 공통 신청서 밖으로 빼지 않는다.

- `과목 확인 필요(migration_review)`의 귀속·목적지 확인은 문의 정보 섹션에 둔다.
- `문의 완료`의 재개는 문의 정보 섹션, `미등록 완료`의 재개는 등록·대기 정보 섹션에 둔다.
- 등록 완료 후 `추가 수업`과 `수강 취소`는 등록·대기 정보 섹션에 둔다. 추가 수업이 입학 처리로 넘어가면 같은 입학 처리 섹션에서 새 배치를 진행한다.
- 예외 행동은 모두 과목명을 표시하고 기존 서버 권한·차단 조건·이력 규칙을 사용한다.

### 7.3 상태별 입력·행동 매트릭스

아래 표는 저장된 과목 트랙 하나의 현재 상태가 어떤 **워크플로 소유 값과 행동**을 여는지 정한다. 학생·보호자·요청 정보와 과목 구성은 이 표와 무관하게 6.2의 기존 잠금·동기화 규칙을 따른다. 영어와 수학의 상태가 다르면 같은 신청서에서 각 과목에 해당하는 입력과 행동이 동시에 열릴 수 있다. 각 과목에는 현재 하위 단계의 주 행동 하나만 강조하고, 취소·재개·경로 변경은 보조 행동으로 둔다.

| 과목 상태 | 입력 가능한 섹션·값 | 명시적 행동 |
| --- | --- | --- |
| `문의` | 다음 경로에 필요한 레벨테스트 예약·상담 담당·대기 값 | 레벨테스트 예약, 바로 전화상담, 대기 전환, 문의 종료 |
| `과목 확인 필요` | 문의 정보의 과목 귀속·목적지 | 과목 확인 완료 |
| `레벨테스트 예약` | 레벨테스트 예약과 아직 처리되지 않은 응시 값 | 예약 변경, 시험 시작, 결과 기록, 예약 취소, 허용 조건 충족 시 문의 종료 |
| `레벨테스트 진행` | 레벨테스트 결과 | 결과 기록 |
| `전화상담 대기` | 상담 책임자와 상담 결과 | 전화상담 완료, 방문상담 예약 |
| `방문상담 예약` | 방문상담 예약과 상담 결과 | 예약 변경·취소, 방문상담 완료 |
| `대기` | 대기 종류·수업, 재응시 판단 | 대기 변경, 레벨테스트 재응시, 등록 전환, 미등록 종료 |
| `등록 결정` | 수강 수업·교재·시작 일정 | 입학신청서 발송, 입학 처리 시작, 대기 전환, 미등록 종료 |
| `등록 처리` | 입학 처리 체크리스트 | 현재 입학 단계 실행, 등록 완료, 입학 처리 취소 |
| `등록 완료` | 추가 수업 초안 | 추가 수업 시작, 수강 취소 |
| `미등록 완료` | 기존 값은 읽기 전용 | 재개 |
| `문의 완료` | 기존 값은 읽기 전용 | 재개 |

표의 행동은 화면 표시 위치를 정의한다. 실제 실행 가능 여부와 필수값은 현재 `ALLOWED_ACTIONS_BY_STATUS`, 전이 차단 조건, 권한 검사와 서버 RPC가 계속 권위 계약이다. 이전 단계의 확정된 워크플로 값은 계속 보이지만 일반 입력으로 되돌려 쓰지 않고 해당 변경·취소 행동을 사용한다.

## 8. 저장과 데이터 흐름

### 8.1 섹션별 커밋

각 섹션은 **과목별 현재 하위 단계마다** 하나의 명확한 주 행동만 강조한다. 영어와 수학의 단계가 다르면 한 섹션 또는 서로 다른 섹션에서 과목별 주 행동이 각각 하나씩 활성화될 수 있다. 다만 공유 예약, 신청 건 단위 입학신청서 발송, 여러 과목을 묶은 입학 처리 배치는 중복 버튼을 만들지 않고, 참여 과목 배지가 붙은 하나의 공유 행동으로 표시한다.

- 문의 정보 저장
- 예약 저장 또는 결과 저장
- 상담 결과 저장
- 등록·대기 정보 저장
- 입학 처리의 현재 단계 실행

한 섹션을 저장하는 동안 다른 섹션의 저장 상태를 덮어쓰지 않는다. 저장하지 않은 값이 있는 섹션만 닫기 경고 대상이 된다.

### 8.2 정규화된 데이터 모델 유지

데이터베이스를 노션처럼 한 테이블로 합치지 않는다.

현재의 부모 신청 건, 과목 트랙, 예약, 레벨테스트, 상담, 수강, 입학 배치, 이벤트 테이블을 권위 데이터로 유지한다. 서비스 계층이 이를 `OpsRegistrationCaseDetail` 형태로 조합하고 공통 신청서가 한 문서로 렌더링한다.

이 설계에 필요한 데이터 변경은 목록을 신청 건 단위로 그룹화하는 조회 계약과 화면 조합이며, 기존 등록 데이터를 다른 형태로 재배치하는 마이그레이션은 기본 범위가 아니다.

### 8.3 상태 보기 조회

목록 서비스는 과목 트랙 행을 그대로 렌더링하지 않고 `taskId` 기준으로 묶은 신청 건 요약을 반환한다.

요약에는 다음 값이 포함된다.

- 신청 건 식별자와 공통 학생 정보
- 과목별 트랙 식별자와 상태
- 현재 보기와 일치하는 과목
- 현재 보기와 일치하는 과목별 다음 처리 시각과 담당자
- 현재 보기의 기존 트랙 정렬 규칙으로 계산한 대표 정렬값
- 관련 예약 식별자

기존 과목별 상태 계산과 정렬 기준은 유지하되 화면 행만 신청 건 단위로 중복 제거한다.

## 9. 컴포넌트 경계

현재 `registration-track-editor.tsx`의 책임이 크므로 공통 신청서를 작은 단위로 분리한다.

- `RegistrationApplicationShell`: 신규·상세에 공통인 문서 틀과 닫기/새로고침 상태
- `RegistrationApplicationHeader`: 학생과 과목별 상태 요약
- `RegistrationInquirySection`: 공통 정보와 과목 구성
- `RegistrationLevelTestSection`: 공유 예약과 과목별 응시·결과
- `RegistrationConsultationSection`: 과목별 담당자, 전화 대기, 방문 예약, 결과
- `RegistrationPlacementSection`: 대기·수업·교재·시작 일정
- `RegistrationAdmissionSection`: 신청 건 단위 입학 처리
- `RegistrationHistorySection`: 읽기 전용 자동 이력
- `RegistrationCaseListRow`: 신청 건 하나와 과목별 상태를 표시하는 목록 행

기존 서비스 함수와 RPC는 각 섹션이 호출하는 권위 인터페이스로 유지한다. UI 통합을 이유로 서로 다른 업무 변경을 하나의 거대한 저장 함수에 합치지 않는다.

## 10. 오류와 충돌 처리

1. **검증 오류:** 해당 섹션 안에서 한국어로 표시하고 첫 오류 필드에 초점을 둔다. 다른 섹션으로 이동하는 팝업을 열지 않는다.
2. **권한·선택 데이터 미준비:** 영향을 받는 섹션만 읽기 전용으로 만들고 `다시 불러오기`를 제공한다. 이미 입력 가능한 다른 섹션은 막지 않는다.
3. **리비전 충돌:** 최신 서버 값을 다시 불러왔음을 명확히 알리고 사용자가 차이를 확인한 뒤 다시 저장하게 한다.
4. **커밋 성공 후 새로고침 실패:** `저장은 완료됐지만 최신 내용을 불러오지 못했습니다`를 표시하고 같은 요청을 재전송하지 않는다.
5. **알림 후속 처리 실패:** 예약·상담 저장 성공과 알림 실패를 구분한다. 데이터 커밋을 되돌리지 않고 기존 재확인 흐름을 사용한다.
6. **예약 중복·취소:** 날짜가 아니라 예약 식별자와 리비전을 기준으로 처리하고 오래된 알림은 기존 취소 규칙을 따른다.

## 11. 반응형·접근성

- 데스크톱은 관련 필드를 최대 두 열로 배치하고 모바일은 한 열로 유지한다.
- 섹션과 필드의 DOM 순서는 시각 순서와 같아야 한다.
- 영어·수학 입력의 접근 가능한 이름에 과목을 포함한다.
- 빈 값, 잠긴 값, 저장된 값을 색만으로 구분하지 않는다.
- 잠긴 미래 섹션에도 제목과 필드명이 보이며, 섹션당 한 줄의 잠금 사유를 제공한다.
- 문서 내부 단계 탭, 설명용 요약 카드, 중복된 `현재 업무` 카드는 추가하지 않는다.
- 모바일에서도 섹션의 주 행동이 입력 필드 다음에 자연스럽게 오도록 한다.

## 12. 알림 안전 경계

이번 UI 변경은 Google Chat, Web Push, SOLAPI 또는 알림 worker의 활성화 권한을 포함하지 않는다.

- 기존 outbox, `appointmentId`, 알림 리비전, 중복 방지 키를 유지한다.
- 브라우저 검증에서는 모든 실제 발송 공급자를 비활성 상태로 둔다.
- UI가 한 페이지로 합쳐져도 저장 이벤트 수나 알림 이벤트 의미를 임의로 합치지 않는다.
- 알림 전송 준비 여부는 저장 성공 여부와 별도 상태로 보여 준다.

## 13. 테스트 전략

### 13.1 계약·단위 테스트

- 신규와 상세가 같은 여섯 섹션 순서를 렌더링한다.
- 미래 섹션은 숨지 않고 빈 값과 잠금 사유를 표시한다.
- 상태 전환에 따라 필요한 필드만 잠금 해제된다.
- 상태별 입력·행동 매트릭스와 서버 전이 차단 조건이 일치한다.
- 영어·수학 값과 상태가 서로 독립적이다.
- 공유 예약은 한 편집 영역과 여러 과목 배지로 나타난다.
- 두 과목이 모두 등록 결정이어도 입학신청서 발송 행동은 신청 건당 하나만 렌더링된다.
- 목록은 보기 안에서 신청 건당 한 행만 반환한다.
- 같은 보기에 두 과목이 있으면 두 과목의 담당자·시각을 보존하고 대표 트랙 규칙으로 안정 정렬한다.
- 한 신청 건이 서로 다른 과목 상태 때문에 여러 상태 보기에 포함될 수 있다.
- 완료 보기는 모든 과목 트랙이 종료 상태일 때만 포함한다.
- `현재 업무` 외부 카드와 신청서 밖 일정 편집기가 렌더링되지 않는다.
- 과목 확인, 종료 재개, 추가 수업, 수강·입학 처리 취소가 지정된 신청서 섹션 안에 있다.
- 자동 이력에는 수정·삭제 입력이 없다.

### 13.2 서비스·데이터베이스 회귀 테스트

- 기존 원자적 생성, 리비전 충돌, 요청 키 멱등성 계약을 유지한다.
- 레벨테스트·방문상담 공유 예약과 과목 참여 관계를 유지한다.
- 상담 완료, 대기, 등록, 미등록, 취소·재개 전환이 기존 이벤트를 남긴다.
- 입학 처리 메시지의 불확실한 결과를 자동 재전송하지 않는다.
- 신청 건 중심 목록 요약이 과목 트랙이나 예약을 누락·중복하지 않는다.

### 13.3 브라우저 검증

로컬 고정 fixture에서 데스크톱과 모바일을 모두 확인한다.

1. 신규 등록을 열면 전체 공통 신청서가 보인다.
2. 문의 저장 뒤 창이 닫히지 않고 같은 신청서가 저장된 상태로 전환된다.
3. 영어와 수학을 서로 다른 단계로 진행해도 한 신청서에 남는다.
4. 상태 탭을 바꿔도 같은 신청 건이 같은 신청서를 연다.
5. 예약, 시험 결과, 상담 결과, 대기, 수강, 입학 처리가 각 섹션 안에서 완료된다.
6. 새로고침 후 모든 값과 현재 단계가 복원된다.
7. 자동 이력에 행위자와 시각이 나타나며 직접 편집할 수 없다.
8. 브라우저 콘솔 오류가 없고 실제 알림 공급자 호출이 0건이다.

## 14. 완료 기준

- 등록 생성과 상세가 동일한 공통 신청서 구조를 사용한다.
- 문의부터 자동 이력까지 모든 섹션이 항상 한 페이지에 존재한다.
- 미래 단계의 빈 값이 보이고 선행 조건에 따라 잠금 상태가 바뀐다.
- 실제 입력과 업무 행동이 해당 신청서 섹션 안에 있다.
- 영어·수학이 독립적으로 진행되면서도 학생 신청 문서는 하나다.
- 목록의 한 행과 건수는 상태 보기 안에서 고유 신청 건 기준이다.
- 기존 데이터 모델, 멱등성, 충돌 처리, 예약·알림 안전 규칙이 유지된다.
- 자동 이력은 읽기 전용이고 책임자와 시각은 자동 기록된다.
- 데스크톱·모바일 고정 fixture와 관련 회귀 테스트가 통과한다.

## 15. 범위 밖

- 등록 데이터를 하나의 비정규화 테이블로 합치기
- `문의 채널` 필드 재도입
- 전반·퇴원·휴보강 자체의 재설계
- 실제 알림 공급자 활성화 또는 운영 cutover
- 운영 데이터로 저장·발송 시나리오 시험
- 모든 섹션을 한 번에 저장하는 전역 저장 버튼
- 현재 등록 업무와 무관한 공통 컴포넌트 리팩터링
